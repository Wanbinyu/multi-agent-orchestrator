# 任务书：将 Multi-Agent Orchestrator 进化为实用 Agent 工具

> 状态：Phase 1 CLI 版连接向导已实现，正在推进图形化配置界面  
> 后续输入：仍可学习 OpenCode 开源项目源码（已获作者许可），但当前先从独立 UI 方案入手

---

## 一、目标与愿景

把当前项目从一个“需要手动配置、固定流程、终端交互的多 Agent 编排器”，逐步升级为一个**像 Claude Code 一样好用、但支持多模型协同**的实用 Agent 工具。

最终形态参考：

- 连接模型像 CCswitch 一样简单（一键配置、自动发现可用模型）。
- 交互方式像 Claude Code 一样自然（对话式输入任务，而不是选择固定任务类型）。
- 工具能力覆盖本地文件系统（读文件、写文件、执行命令、浏览目录）。
- 主模型能感知已连接的其他模型，并根据模型特点分配子任务。
- 整个体验更偏向“一个能干活的多模型编程/创作助手”，而不是“一个需要预先定义流水线的编排框架”。

---

## 二、核心理念

### 1. 连接优先

现在的痛点：配置 `providers.yaml`、设置 API Key、选择模型名太繁琐。

目标状态：

- 打开应用 → 添加 Provider（火山方舟 / OpenAI / Anthropic / Kimi 转发 / 自定义）。
- 输入 API Key 或 Base URL，自动测试连通性。
- 连通后自动列出可用模型，用户只需勾选/设置主模型。
- 所有已连通模型进入“模型池”，后续主模型可以调用它们。

### 2. 对话即任务

现在的痛点：用户必须给 `run.py` 传一个“需求字符串”，系统按固定流程拆任务。

目标状态：

- 用户在聊天框里像和 Claude 对话一样描述任务。
- 主模型自己判断：这个任务需要拆吗？需要哪些工具？需要调用其他模型吗？
- 不需要用户先选“软件开发”还是“小说创作”场景。

### 3. 模型-aware 的任务分配

现在的痛点：Orchestrator 硬编码拆任务，Worker 模型名写死。

目标状态（两种模式可选）：

#### 模式 A：自动分配

- 主模型根据已连接模型的名字/能力标签，自动判断谁适合干什么。
- 例如：
  - `claude-opus-*` / `glm-*` → 复杂推理、架构设计
  - `claude-sonnet-*` / `kimi-*` → 代码实现、正文写作
  - `claude-haiku-*` / `glm-lite-*` → 快速检查、简单摘要

#### 模式 B：人工确认分配

- 主模型先把任务拆成若干子工作。
- 然后弹出一个面板，让用户选择“这部分用哪个已连接模型做”。
- 适合对结果质量敏感、想控制成本的用户。

### 4. 本地工具能力

现在的痛点：Worker 只能写文件，读文件/执行命令受限，且无法主动浏览目录。

目标状态：

- 集成完整的本地工具集：
  - `read_file`：读文件
  - `write_file`：写文件
  - `list_dir`：浏览目录
  - `run_command`：执行命令（带白名单/确认机制）
  - `search_files`：文件内容搜索
  - `edit_file`：局部修改文件
- 支持读取用户电脑上的项目文件夹（需要系统权限弹窗/沙箱机制）。

## 三、主要功能方向

### Phase 1：连接层简化

- 图形化或 TUI 配置界面，替代手动编辑 YAML。
- Provider 自动连通检测：添加后自动发一个测试请求。
- 模型池：连通后自动枚举可用模型。
- 主模型选择：从模型池里选一个默认主模型。

### Phase 2：交互层改造

- 把命令行 `python run.py "需求"` 改为持续对话界面。
- 支持多轮对话，主模型记住上下文。
- 用户可以随时追加需求、纠正方向。

### Phase 3：工具能力补齐

- 文件系统工具：读、写、列目录、搜索、局部编辑。
- 命令执行工具：白名单 + 超时 + 用户确认。
- 工具调用循环：模型可以根据执行结果再次调用工具，直到任务完成。

### Phase 4：多模型协作

- 主模型可以决定是否需要调用其他模型。
- 子模型调用结果返回给主模型，主模型继续主导整体进度。
- 支持“自动分配”和“人工确认分配”两种模式。

### Phase 5：体验打磨

- 更好的 UI（Web 或 TUI）。
- 任务进度可视化。
- 成本实时显示。
- 错误处理和重试策略。
- 可导出会话记录和最终产出。

## 四、技术架构方向

```text
┌─────────────────────────────────────────┐
│              UI 层（对话界面）            │
│      Web / TUI / VS Code Extension      │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│            Session / 上下文管理          │
│   多轮对话、工具调用历史、文件快照        │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│            主模型（Main Agent）          │
│   理解需求、规划任务、调用工具/子模型     │
└─────────────────────────────────────────┘
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
┌─────────┐    ┌─────────┐    ┌─────────┐
│  工具层  │    │ 子模型层 │    │ 文件系统 │
│ Tools   │    │ Workers │    │  FS     │
└─────────┘    └─────────┘    └─────────┘
    │               │               │
┌─────────────────────────────────────────┐
│         Provider / Gateway 层           │
│   统一封装多 Provider 调用、计费、重试    │
└─────────────────────────────────────────┘
```

### 关键变化

| 当前架构 | 目标架构 |
|----------|----------|
| Orchestrator 固定拆任务 | 主模型动态规划，可拆可不拆 |
| Dispatcher 并行执行所有 Worker | 主模型按需调用子模型或工具 |
| Reviewer 最后统一审查 | 主模型在过程中自我纠错、验证 |
| 终端命令式 | 对话式交互 |
| 手动 YAML 配置 | 图形/TUI 配置 + 自动连通检测 |
