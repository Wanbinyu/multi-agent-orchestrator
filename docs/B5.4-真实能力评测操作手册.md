# B5.4 真实能力评测操作手册

**状态**：离线框架与 Harbor adapter 已完成；private 小样本已运行，多模型真实验收暂停

**目标**：在相同任务、工作区、执行深度和验收器下，分别测量固定单模型、自动路由单 Agent 和多模型协作。

## 1. 当前完成的合同

- `fixed-single`：固定主模型，禁用 Worker，用作单模型基线。
- `auto-route`：允许可解释路由，禁用 Worker，只测路由价值。
- `multi-model`：允许可解释路由并强制合格的工程任务进入 Orchestrator/Worker/Reviewer，只用 `standard/deep`。
- `mao benchmark-agent`：无交互执行生产 Agent 流程，输出 JSON 中的路由、协作、轨迹、token、成本、文件和工程审计事实。
- `MaoHarborAgent`：符合 Harbor 当前 `BaseInstalledAgent` 接口，用 `MAO_INSTALL_SPEC` 锁定 wheel 或 Git commit，用脱敏 Provider 模板在容器内运行 MAO。
- 所有策略都有 `allowed_models`；固定策略自动只允许主模型，自动路由/多模型策略必须明确至少两个候选。未授权模型会在调用前从 Gateway 候选和 failover 链排除。

离线验收命令：

```bash
python -m pytest -q tests/test_benchmark_agent.py tests/test_engineering_benchmark.py
python scripts/benchmark_engineering.py
```

第二条命令对 6 类程序化任务运行 3 种策略、3 次重复，合计 54 条合成结果。它只验证控制变量、隔离、报告和稳定性，`provider_calls=0`，不得作为真实模型效果结论。

## 2. 真实调用前的必填授权单

需要项目所有者一次性确认：

1. `fixed-single` 的模型别名。
2. `auto-route` 允许的模型别名列表。
3. `multi-model` 允许的模型列表，以及 workers 配置中各角色用哪个模型。
4. 每个任务/策略的重复次数，以及 Provider 尝试次数停止阈值。该次数在每个真实网络请求前预占，达到阈值后的请求不得发出。
5. 本地费用停止阈值和 Provider 账号侧硬额度。本地只能在返回 usage 后计费，单次请求可能越过阈值，因此真正的硬上限必须在 Provider 侧配置。
6. 结果可见性：`private`、`aggregate` 或 `full`。

代码层使用 `LiveBenchmarkAuthorization` 和所有策略共享的 `LiveBenchmarkSpendGuard`。没有授权对象时，真实 runner 不应被创建。费用仍是 Provider 返回 usage 后才能确认的后付费指标；次数是本地硬上限，费用的真正硬上限仍需在 Provider 账号侧配置。

## 3. Harbor/Terminal-Bench 准备

当前适配目标是 [Harbor](https://harborframework.com/docs/agents) `0.20.x` 和 [Terminal-Bench 2.0](https://www.tbench.ai/)。Harbor `0.20.x` 要求 Python 3.12；MAO 默认安装仍支持 Python 3.11/3.12，只有 `[benchmark]` 评测环境必须使用 3.12。Harbor 不放入 MAO 默认依赖：

```bash
python -m pip install -e ".[test,benchmark]"
harbor run --help
```

`MAO_INSTALL_SPEC` 必须锁定可重现版本，正式数据禁止使用浮动的 `main`/`latest`：

```text
git+https://github.com/Wanbinyu/multi-agent-orchestrator.git@<full-commit-sha>
```

Provider 模板的 `api_keys` 只写环境变量引用，例如 `${ARK_API_KEY}`。不要把真实 Key 写入 YAML 再做 Base64；Base64 不是加密。真实 Key 通过 Harbor `--ae` 注入，模板通过 `MAO_PROVIDERS_CONFIG_B64` 注入。

示例策略参数：

```bash
--agent src.integrations.harbor_agent:MaoHarborAgent
--ak strategy=fixed-single --ak main_model=glm-ark --ak allowed_models=glm-ark
--ak execution_depth=standard --n-concurrent 1
```

自动路由改为 `strategy=auto-route`，多模型改为 `strategy=multi-model`；后两者的 `allowed_models` 使用逗号分隔的至少两个别名。报告会单独保存 `actual_models`，不用策略标签代替实际调用事实。首轮只运行一个公开任务、串行、一次重复，检查容器写入、verifier、`mao-result.json`、轨迹和费用后再扩大。

## 4. 结果解读约束

- 完成率以相同确定性 verifier 为准，不以模型自述“已完成”为准。
- token/成本结论必须标明 Provider 是否返回真实 usage，未知价格不生成节省声明。
- 模型、执行深度、路由和协作是四个独立变量；报告不得把它们混成一个“MAO 更强”结论。
- 失败结果必须保留。如果多模型没有改善某类任务，应当记录为路由边界，不删除数据。
- Terminal-Bench 任务原文、私有项目、Key 和不可再分发产物不进入公开报告。

## 5. 2026-07-19 private smoke 记录

- 授权模型：`glm-ark`、`kimi-for-coding`；累计次数上限从 20 提高到 35；费用上限 `$0.20`；结果可见性 `private`。

- `multi-model` 故障收敛修复了顶层 JSON 任务数组、未注册角色别名、列表型 `acceptance/output_format`，以及软件任务被误分配给创作 `writer` 的问题。
- 旧停止门只在整个 Agent 返回后执行 `record`。最后一轮在本轮上限 8 次时实际产生 13 次，累计达到 **40/35**、累计真实成本 `$0.032453`。这是停止门实现缺陷，不是新的所有者授权。
- 修复后 Gateway 在每个同步、流式和连接测试 Provider 请求前调用线程安全硬预占门，并累计跨回合尝试；达到上限后的第 N+1 次在网络调用前抛错。离线测试已覆盖同步/流式两次允许、第三次 Provider 调用数保持为 2，以及 100 次并发争用只接受授权数量。
- Worker 归一化区分通用角色别名与软件语义漂移：`code_writer` 等别名会保留已配置的合法模型；软件任务误用 `writer/editor` 等创作角色时，才切换到对应软件 Worker 及其默认模型。
