# MAO 手动功能验证指南

> 手动验证工具、Hooks、原生 tool use、上下文、MCP 和本地模型。源码开发命令在项目根目录执行；安装用户可直接使用 `mao`。

---

## 准备

```bash
# 1. 确认测试全绿（v0.1.0-beta.2 基线为 506 passed）
python -m pytest -q

# 2. 启动 CLI 对话（二选一）
mao

# 2. 或启动 Web UI（二选一）
mao web --no-open
# 然后浏览器打开 http://127.0.0.1:8123
```

> 前置：`config/providers.yaml` 已配置好可用模型与主模型（你之前已配置）。CLI 默认用主模型。

已发布 beta.4 的完整测试集合为 `722 passed, 1 warning`；B4.S6 完成时的历史基线为 `683 passed, 1 warning`。高风险前端合同可单独验证：

```bash
python -m pytest -q tests/test_frontend_contract.py tests/test_orchestrator.py tests/test_worker.py tests/test_reviewer.py tests/test_agent_collaboration.py tests/test_registry.py
```

这些测试不调用付费模型，覆盖固定职责、依赖/所有权、缺页与错误 import、真实命令证据、Reviewer 完成门和 RunJournal 多职责记录。

S4 浏览器 smoke 使用 Python Playwright。安装项目依赖后会自动获得 Python 包；运行时优先使用系统 Edge/Chrome。若机器没有可用 Chromium 内核，手工执行一次：

```bash
playwright install chromium
```

离线正反夹具验证：

```bash
python -m pytest -q tests/test_frontend_smoke.py
```

正夹具覆盖登录、七个主路由、表格/画布与两档响应式布局；错误 Mock 登录夹具必须失败并确认 server 已清理。该验证不调用 Provider。

S6 整链离线发布门：

```bash
python -m pytest -q tests/test_delivery_report.py tests/test_stability_replay.py
python scripts/replay_smart_mining.py
```

脚本必须返回 0；输出中的 `good` 为 `completed`，`broken_mock` 与 `missing_route` 为 `blocked`，三者 `provider_calls` 均为 0。它会执行真实本地 npm 命令和浏览器 smoke，但不会调用任何配置模型。

B4.3 会话恢复离线验证：


B4.4 分层压缩验证：


B4.5 增量项目索引验证：

```bash
python -m pytest -q tests/test_project_index.py tests/test_memory.py tests/test_memory_tools.py tests/test_search_tools.py tests/test_read_cache.py
```

首次刷新读取全部可索引文本；第二次必须 `read=0`。只修改一个文件后必须 `read=1`、`updated=1`，其余条目计入 reused；只改 mtime 且内容 hash 相同应计入 metadata_only。损坏 YAML 和项目根切换必须重建，不能返回旧项目搜索结果。

B4.6 Reviewer 信息限制验证：

```bash
python -m pytest -q tests/test_reviewer.py tests/test_agent_collaboration.py tests/test_collaboration_boundaries.py tests/test_engineering_verification.py
```

## 3. Hooks（审计日志）

```bash
# 复制示例配置（去掉 .example 后缀）
copy config\hooks.yaml.example config\hooks.yaml
```

`hooks.yaml` 默认启用审计钩子（`audit_pre`/`audit_post`），把每次工具调用写入 `logs/tool_audit.log`。

重启 CLI（`python run.py chat`），执行任意工具调用，然后：
```bash
type logs\tool_audit.log
```
**预期**：日志含 `CALL <工具名>` 与 `OK/ERR <工具名>` 行。

> Hooks 也支持自定义：在 `hooks.yaml` 写 `pre`/`post` 列表，每项是可导入的 `模块:函数` 路径。详见 `docs/工具开发指南.md` 第五节。

故障验收：临时写入一个不存在的 Hook 路径并重启。预期 CLI 显示脱敏配置摘要但仍能进入对话；摘要不得出现 Key、环境变量值、完整命令或原始异常。Web 可访问 `http://127.0.0.1:8123/api/diagnostics/extensions` 查看最多 10 条诊断，同时 `/health` 仍返回 `{"status":"ok"}`。

## 4. 原生 tool_use（仿 Claude）

默认 Markdown 模式。启用原生模式：编辑 `config/providers.yaml`，在主模型配置下加一行：

```yaml
models:
  glm-ark:                      # 你的主模型
    provider: glm
    model_id: glm-4.6
    # ... 其它字段
    native_tools: true          # 加这一行，强制启用原生 tool_use
    # 或：在 capabilities 里加 tool_use，也会自动启用
    # capabilities: [tool_use, coding, reasoning]
```

重启 CLI，发一个需要工具的任务（如"读取 README.md"）。

**如何确认原生模式生效**：
- 打开当前会话 YAML（`sessions/<会话id>.yaml`），看第一条 system 消息：
  - **原生模式**：不含 ```` ```tool: ```` 代码块（工具定义走 `tools=` 参数）。
  - **Markdown 模式**：含 ```` ```tool:read_file ```` 等工具块。
- 行为差异：原生模式下模型用结构化函数调用，工具调用更稳定。

> 注意：模型本身需支持 function calling（Claude / GLM-4.6 / Kimi 等支持）。不支持的模型保持 Markdown 兜底，仍可用。

## 5. 自动压缩（长会话不中断）

为便于观察，人为调低触发阈值。编辑 `config/providers.yaml` 主模型：
```yaml
  glm-ark:
    # ...
    max_context_tokens: 8000   # 调低，约 6000 token 时触发压缩
```

进 CLI，制造长对话：
- 多轮来回（10+ 轮），或让模型读取大文件（`读取 src/gateway/provider.py` 这类长文件）。
- 继续对话直到接近阈值。

**预期**：
- 对话不报"上下文超长"错误，能持续。
- （可选）在代码里临时加日志观察：`src/core/agent.py` 的 `_maybe_compact_context` 返回 True 时即触发了压缩。

> 验证完记得把 `max_context_tokens` 改回真实值（如 GLM-4.6 用 131072），否则会过早压缩浪费 token。

## 6. MCP 外部工具（可选，需安装）

`mcp.yaml` 默认配了 `@modelcontextprotocol/server-filesystem`（需 Node.js/npx）。确保 npx 可用：

**预期**：列表里出现 MCP server 暴露的工具（如 `read_file`、`write_file` 等来自 filesystem server，可能与内置同名--内置优先）。

调用一个 MCP 独有工具验证。若 server 启动失败，检查 `mcp.yaml` 的 command/args。

> MCP 适配器代码和 mock 测试已经完成；本节验证的是可选依赖、Node 环境与具体 Server 的真实连通性。

## 7. 本地 LLM（可选）

### 方式 A：Ollama（推荐）
1. 安装 Ollama：https://ollama.com
2. 拉模型：`ollama pull qwen2.5:7b`
3. 启动 Ollama 服务（默认 11434 端口）
4. 在 `config/providers.yaml` 加（参照 `config/providers.yaml.example`）：
   ```yaml
   providers:
     ollama:
       name: ollama
       type: ollama
       base_url: http://localhost:11434/v1
       api_keys: []
       timeout: 300
   models:
     qwen-local:
       provider: ollama
       model_id: qwen2.5:7b
       capabilities: [coding]
       max_context_tokens: 32768
   main_model: qwen-local   # 切为主模型
   ```
5. 重启 CLI，对话即用本地模型（计费显示 $0）。

### 方式 B：llama.cpp 进程内
```bash
pip install llama-cpp-python
```
下载 GGUF 模型，在 providers.yaml 配 `type: llamacpp`，`base_url` 填 GGUF 路径（见 example）。

## 常见问题

| 现象 | 排查 |
|---|---|
| `/tools` 不见新工具 | 确认 `src/tools/worker_tools.py` 顶部 import 了对应工具模块 |
| 工具调用被拒 | 当前是 `readonly` 或 `approve` 未确认；`/auto` 切自动 |
| 原生模式没生效 | 模型不支持 function calling；或 `native_tools` 没加到主模型 |
| MCP 工具不出现 | `pip install mcp`；检查 `config/mcp.yaml`；看 npx 是否能拉起 server |
| Web UI 工具不显示 | Web 启动时 lifespan 加载扩展；确认 `config/` 下有对应 yaml |
