# MAO 开源发布准备计划

**状态**：开源发布门已通过（远端 CI 全绿）；待创建 `v0.1.0-beta.1` Release，公开可见性需所有者单独确认

**目标版本**：`v0.1.0-beta.1`

**开源触发点**：Phase 7.4 完成后立即提醒项目所有者进入开源验收；通过本计划的发布门后再公开仓库和创建 GitHub Release。

## 一、发布定位

首个公开版本定位为可实际使用、核心流程可信、仍在快速演进的 Beta，不宣传为 Claude Code、Codex 或其他成熟产品的完整替代品。

核心承诺：

- 支持多 Provider、多模型编排和国内模型接入。
- 解释、诊断、修改、审查等任务具有明确授权边界。
- 工具调用、文件修改、工程记录和验证结果可追踪。
- 模型或 Provider 失败时有明确状态，不伪装成成功。
- 上下文窗口和压缩能力按真实配置说明，不猜测模型身份或上限。

首个版本不承诺：

- 突破上游模型上下文硬限制。
- 所有模型都稳定支持原生工具调用。
- 完整容器级代码沙箱。
- VS Code、Electron 或独立可执行文件等所有分发形态。
- 在所有任务上优于单模型 Agent。

## 二、推荐开源阶段

```text
Phase 7.0 运行状态基础（已完成）
  -> Phase 7.1 任务分类与执行策略
  -> Phase 7.2 证据与假设循环
  -> Phase 7.3 验证门与完成审计
  -> Phase 7.4 有边界的多模型工程协作
  -> 开源提醒 + 发布验收迭代
  -> v0.1.0-beta.1
```

不等待 Phase 7.5-7.6 全部完成。Phase 7.4 完成时，MAO 的核心故事应当形成闭环：知道用户要什么、按边界执行、保留证据、完成验证，并能让多个模型在可控范围内协作。

## 三、Phase 7.4 前的核心门槛

### 1. Phase 7.1：任务类型与安全边界

- 支持 answer、explain、diagnose、change、build、review、plan、monitor。
- 解释和诊断默认只读，除非用户明确要求修复。
- 修改和构建必须记录写入授权、作用域和预期交付物。
- 分类失败时采用保守策略，不自动扩大权限。

**验收**：常见自然语言请求能稳定落入正确策略；只读任务不会写文件。

### 2. Phase 7.2：证据驱动执行

**状态**：已于 2026-07-15 完成；通过 `449 passed` 全量回归和真实 Web SSE 异常路径验证。

- 陌生项目先显示结构、Git 状态、入口、依赖和测试位置。
- 工具结果转换为可引用 Evidence。
- 区分事实、假设、反证和未检查区域。
- 抑制重复读取和无界项目扫描。

**验收**：项目审查结论能追溯到文件、工具结果或测试，而不是只依赖模型表述。

### 3. Phase 7.3：验证门与完成审计

**状态**：已于 2026-07-15 完成；通过 `469 passed` 全量回归和真实 Web SSE 异常路径验证。

- 按风险选择针对性测试、相邻回归、全量回归或手动烟雾验证。
- 需求、实现和验证形成对应关系。
- 缺少关键验证时禁止宣称“已完成”。
- Reviewer 使用证据审查，而不是重新自由猜测。

**验收**：完成状态能够证明；失败和剩余风险明确可见。

### 4. Phase 7.4：有边界的多模型协作

**状态**：已于 2026-07-15 完成；通过 `485 passed` 全量回归和真实 Web SSE 异常路径验证。

- 子任务具有完整输入、输出格式、验收标准和文件所有权。
- 只对互不依赖或可隔离的任务并行。
- Worker 失败可定位到具体任务和模型，并支持定向重试。
- Reviewer 能基于各 Worker 证据完成收口。
- 多模型协作遵循与单 Agent 相同的权限和验证边界。

**验收**：复杂工程任务不会因并行造成文件冲突、重复执行或无证据合并。

## 四、上下文稳定性门槛

开源前至少完成《上下文扩展与长任务稳定性计划》的 Context 1-2：

- 模型窗口、最大输出和预算来源可解释。
- 未知模型保持 32K 保守预算并标记未验证。
- Agent、Worker、Reviewer 使用统一的 ContextBudgetManager。
- WebUI 可配置并查看上下文预算与压缩阈值。
- 建立最小长会话基准，覆盖大工具输出、一次压缩和会话恢复。

Context 3-5 的完整分层压缩、项目上下文复用和高级可观察性可以在 Beta 后继续；但公开前不能存在已知的上下文溢出导致 Session 损坏问题。

## 五、发布准备工作流

### A. 安装与分发

- 增加 `pyproject.toml`、正式版本号和 CLI 入口。
- 保留源码运行方式，并提供虚拟环境安装命令。
- 全新 Windows、Ubuntu 环境从克隆到第一次成功对话不超过十分钟。
- Provider、MCP、Hooks 和 Worker 均提供不含密钥的示例配置。
- 创建 GitHub Release，而不只推送 `main`。

### B. CI 与质量门

- GitHub Actions 覆盖 Windows、Ubuntu、Python 3.11/3.12。
- 自动运行 pytest、compileall、JavaScript 语法和 `git diff --check`。
- 增加依赖与常见敏感信息扫描。
- 发布分支不得存在 P0/P1 已知故障。
- CI 不调用真实付费模型；真实模型仅做有记录的人工烟雾验证。

### C. README 与演示

- 第一屏用一句话说明产品、目标用户和核心价值。
- 不再使用已经过时的“CLI MVP”定位。
- 增加 WebUI 截图和 60-90 秒演示 GIF/视频。
- 五分钟快速开始放在架构细节之前。
- 展示一次完整工程任务：结构、权限、工具、协作、验证、成本和最终结果。
- 列出支持的 Provider、操作系统、已知限制和安全边界。
- 提供多模型与单模型的 token、成本、调用数和完成率基准，避免只做营销表述。

### D. 开源治理

- 增加 `CONTRIBUTING.md`。
- 增加 `SECURITY.md`，说明 Key 存储、命令白名单、权限模式和沙箱限制。
- 增加 Issue、Bug、Provider 兼容问题模板。
- 增加 `CHANGELOG.md` 和版本策略。
- 明确维护范围、兼容 Python 版本和 Breaking Change 处理方式。

### E. 安全与隐私

- 确认 `.env`、真实 Provider 配置、Session、Memory 和私人笔记不被跟踪。
- 扫描历史提交中的 API Key、Token、个人路径和真实会话内容。
- README 明确提示使用者不要提交 Key。
- 默认使用 `approve` 或更保守模式；危险能力不得静默开启。
- 容器级沙箱未完成时，在 README 和 SECURITY 中明确本地信任边界。

### F. 真实验收场景

至少验证：

1. 解释一个陌生项目，不修改文件。
2. 诊断一个故障，输出根因、证据和未验证区域。
3. 修改一个小功能，展示计划、差异和针对性测试。
4. 多模型完成一个可拆分任务，不产生文件冲突。
5. 模型 401、429、流中断和 Reviewer 未通过路径正确收口。
6. 长会话接近预算时可以压缩并继续，不损坏会话。

每个场景保存环境、模型、输入、输出、工具调用数、token、成本、验证结果和残余风险。

## 七、Phase 7.4 完成提醒协议

Phase 7.4 被标记为完成时，必须在同一次工作收尾中执行以下动作：

1. 明确提醒项目所有者：**“Phase 7.4 已完成，可以开始开源发布验收。”**
2. 将 `docs/项目开发进度与路线图.md` 的发布状态更新为“待开源验收”。
3. 打开本计划的“发布验收清单”，逐项检查，不直接公开仓库。
4. 汇总仍未满足的阻塞项，并按 P0/P1 排序。
5. 通过全部发布门后，再建议创建 `v0.1.0-beta.1` 和公开仓库。

该提醒由项目里程碑触发，不使用日期型自动提醒。当前自动提醒机制不能可靠判断 Git 中的 Phase 7.4 是否已经完成。
