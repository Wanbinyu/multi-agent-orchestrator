# Phase 6.6：项目结构展示与 CLI 结果可见性

**状态**：进行中（P0 已完成，P1/P2 待实施）
**提出时间**：2026-07-15
**来源场景**：CLI 检查 `G:\MAO_test`，模型读取约 30 个文件后只显示工具调用和 `response.md` 路径，最终审查报告与项目目录树未直接显示在控制台。

## 已完成的相关 CLI 体验改进

- 启动欢迎页不再展开全部命令。
- 输入 `/` 后显示命令候选及说明，继续输入字母实时过滤。
- `/help` 仍可查看完整命令，Shift+Tab 权限模式切换保持不变。
- 工具执行过程新增 `tool_start` / `tool_complete` 流式事件。
- 工具调用按阶段展示，每阶段最多展开 4 项，其余折叠并在结束时统计。
- 单 Agent 工具循环完成后会重新渲染最终答案，不再只显示 `response.md`。
- 历史未命名会话在 WebUI 中显示为“未命名会话”，不再直接展示长 ID。

## 一、修复前现状与根因

### 1. 最终回答被隐藏

`src/cli/chat_command.py::_stream_turn()` 在检测到工具块后会停止实时渲染原始内容。收到 `done` 时虽然取得 `assistant_message`，但修复前只对多模型协作分支打印“最终答案”面板。

结果：单 Agent 工具循环完成后，最终报告已经生成并保存到 `response.md`，控制台却只显示工具调用、文件路径和 token 统计。该问题已由 P0 修复。

### 2. 没有真正的项目树能力

- `list_dir` 只列单层目录，模型需要反复调用才能理解项目。
- `glob_files` 能列文件，但不是便于人阅读的层级树。
- 记忆分类 `project_structure` 只用于长期记忆标签，不提供 CLI 项目结构视图。
- 当前没有 `/tree` 命令、`project_tree` 工具或 Web 文件树面板。

## 二、目标体验

用户输入：

```text
现在在 G:\MAO_test 中有一个项目，帮我检查一下
```

控制台应至少显示：

```text
项目结构：G:\MAO_test
├── README.md
├── index.html
├── backend/
│   ├── requirements.txt
│   └── app/
│       ├── main.py
│       ├── core/
│       ├── models/
│       └── routers/
└── ...

项目概览
严重问题
重要问题
修复建议
```

完整报告仍可保存到 `response.md`，但保存文件不能替代控制台输出。

## 三、实施顺序

### P0：修复 CLI 最终回答可见性 ✅

- 非协作工具循环收到 `done` 后，始终渲染最终 `assistant_message`。
- 简单无工具流式回答保持当前体验，避免重复打印。
- 工具调用列表、最终答案、写入文件和 token 统计按固定顺序展示。
- 最终答案使用独立面板；超长内容必须明确截断或分页，不能静默消失。

实际实现：

- `ChatStreamEvent` 新增 `tool_start` / `tool_complete` 和 `tool_call` payload。
- Agent 在工具执行前后发送结构化事件；失败、只读拒绝和参数错误进入统一失败摘要。
- CLI 使用“探索项目 / 检索代码 / 查询资料 / 生成交付物 / 执行验证”阶段标题。
- 重复调用不再逐条刷屏，但会在“本轮工作”面板中报告折叠数量。
- 结果展示顺序固定为：过程进度 -> 最终答案 -> 本轮工作 -> 交付文件 -> 模型与 token。
- 新增 `tests/test_chat_command_output.py`，锁定“使用工具后最终答案可见”的行为。

### P1：增加 `project_tree` 工具

建议参数：

```text
path: 项目根目录，支持绝对路径
max_depth: 最大深度，默认 4
max_entries: 最大节点数，默认 300
include_hidden: 是否显示隐藏文件，默认 false
```

默认忽略：`.git`、`node_modules`、`__pycache__`、`.pytest_cache`、`.venv`、`venv`、`dist`、`build`、大型二进制目录。

要求：

- 使用结构化遍历，不能调用平台相关的 `tree` / `dir` / `ls` 命令。
- 排序稳定：目录优先、名称排序。
- 到达深度或数量上限时明确显示“已截断”。
- 遵守现有路径解析与权限模式。
- 返回可供 CLI、Web 和模型上下文复用的文本树。

### P1：增加 CLI `/tree` 命令

```text
/tree G:\MAO_test
/tree G:\MAO_test 5
```

- 本地直接执行，不调用模型、不消耗 token。
- 未传路径时使用当前会话项目根目录或会话输出目录。
- 输出与 `project_tree` 工具一致。

### P1：项目审查提示规范

当用户要求“检查项目、审阅项目、了解项目结构”时：

1. 首先调用一次 `project_tree`。
2. 再读取关键入口、配置、依赖和测试文件，避免无差别读取全部文件。
3. 最终回答必须包含“项目结构”章节，并复用精简目录树。
4. 大项目先给结构和抽样策略，再逐步深入，控制 token 消耗。

### P2：Web 项目树面板

- 在聊天页增加可折叠文件树。
- 支持展开目录、点击读取文件、刷新和忽略规则提示。
- 项目树放在页面工具区域，不嵌套进消息卡片。
- 首期只读，不在树上直接删除或移动文件。

## 四、测试要求

- `project_tree`：深度限制、数量限制、排序、忽略目录、空目录、绝对路径、路径不存在。
- CLI：无工具回答不重复；工具循环后最终回答可见；协作回答仍只打印一次。
- `/tree`：不调用 Gateway，不产生 token 计费。
- 集成场景：用临时项目执行“检查项目”，断言控制台同时包含目录树、审查报告和 token 统计。
- Web：项目树接口返回稳定，超大目录不会阻塞页面。

## 五、验收标准

1. 用户不打开 `response.md` 也能在 CLI 看到完整最终结论。
2. 用户能用 `/tree <路径>` 零 token 查看项目结构。
3. 模型检查项目时先输出精简树，再输出审查结果。
4. 目录超过限制时明确截断，不造成上下文爆炸。
5. Windows、Linux 路径行为一致。
6. 全量 pytest 继续通过，并新增 CLI 捕获输出和工具集成测试。

## 六、建议下一步

P0 已完成。下一步实现 `project_tree` 与零 token `/tree`，并让项目审查提示优先读取结构、入口、依赖和测试文件；之后再做 Web 文件树面板。为降低大型项目分析出现的 68k 级输入 token，应在 Phase 7.2 同步增加单轮只读工具缓存和项目索引复用。
