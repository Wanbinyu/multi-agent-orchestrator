# 步骤 2：Provider 模型映射

## 目标

为 Provider 配置增加 `model_map` 字段，使逻辑模型名可在请求上游前映射为真实模型名。  
这一步主要解决 **CCswitch 式转发服务** 的模型名统一问题：上游接口虽然是 Anthropic 兼容协议，但只接受特定模型名（如把所有 Claude 模型名映射到同一个底层模型）。

---

## 修改的文件

| 文件 | 修改内容 |
|------|----------|
| `src/models/schemas.py` | `ProviderConfig` 增加 `model_map: dict[str, str]` 字段 |
| `src/gateway/provider.py` | `BaseProvider` 增加 `map_model_id()`；`AnthropicProvider` / `OpenAICompatibleProvider` 在请求前调用映射 |
| `config/providers.yaml` | `kimi` provider 增加 `model_map`，把常见 Claude 模型名映射到 `kimi-for-coding` |
| `requirements.txt` | 增加 `pytest`，为后续测试做准备 |
| `tests/conftest.py`（新增） | 把项目根目录加入 `sys.path`，使 `src.*` 导入在测试可用 |
| `tests/test_provider_model_map.py`（新增） | 模型映射逻辑的单元测试 |

## 关键实现说明

### 1. ProviderConfig 增加 model_map

```python
class ProviderConfig(BaseModel):
    """Provider 配置"""
    name: str
    type: Literal["anthropic", "openai"]
    base_url: str
    api_keys: list[str]
    timeout: int = 120
    rpm_limit: int = 60
    model_map: dict[str, str] = Field(
        default_factory=dict,
        description="逻辑模型名 -> 上游真实模型名映射"
    )
```

`model_map` 是可选字段，不配置时行为与之前完全一致。

### 2. BaseProvider.map_model_id

```python
def map_model_id(self, model_id: str) -> str:
    """根据 Provider 的 model_map 把逻辑模型名映射为上游真实模型名"""
    if not self.config.model_map:
        return model_id
    mapped = self.config.model_map.get(model_id)
    if mapped:
        return mapped
    return model_id
```

映射逻辑放在 Provider 基类，保证两种具体 Provider 都能复用。

### 3. 请求前进行映射

**AnthropicProvider** 在 `client.messages.create()` 前：

```python
upstream_model_id = self.map_model_id(model_config.model_id)
response = client.messages.create(
    model=upstream_model_id,
    ...
)
```

**OpenAICompatibleProvider** 在 `client.chat.completions.create()` 中：

```python
response = client.chat.completions.create(
    model=self.map_model_id(model_config.model_id),
    ...
)
```

### 4. 响应中的 model 字段

`ChatResponse.model` 现在返回的是**映射后的上游真实模型名**，便于在日志和计费中看出实际调用了哪个模型。  
计费仍使用逻辑 `model_config` 中配置的价格，因此价格波动需要在配置层保持一致。

### 5. config/providers.yaml 中的 kimi 映射

```yaml
kimi:
  name: Kimi 转发服务（via CCswitch）
  type: anthropic
  base_url: https://api.va11.icu/
  api_keys:
    - ${KIMI_API_KEY}
  timeout: 120
  rpm_limit: 60
  model_map:
    claude-sonnet-5: kimi-for-coding
    claude-sonnet-5-20251001: kimi-for-coding
    claude-opus-4-8: kimi-for-coding
    claude-opus-4-8-20251001: kimi-for-coding
    claude-haiku-4-5: kimi-for-coding
    claude-haiku-4-5-20251001: kimi-for-coding
    claude-fable-5: kimi-for-coding
    claude-fable-5-20251001: kimi-for-coding
```

这样即使 Orchestrator 或 Worker 配置了 `claude-sonnet-5` 并路由到 `kimi` provider，上游实际收到的也是 `kimi-for-coding`。

## 遇到的问题与解决方案

### 问题 1：pytest 未安装

**现象**：运行 `python -m pytest` 报错 `No module named pytest`。

**解决**：把 `pytest>=8.0.0` 加入 `requirements.txt`。本地可先用 `pip install pytest` 安装，或通过 `pip install -r requirements.txt` 统一安装。

### 问题 2：测试中的 `src.*` 导入路径

**现象**：测试文件放在 `tests/` 下，直接 `import src.xxx` 会失败，因为项目没有 `__init__.py`，且测试启动目录不同。

**解决**：新增 `tests/conftest.py`，把项目根目录动态加入 `sys.path`：

```python
project_root = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
sys.path.insert(0, project_root)
```

这是 pytest 的标准做法，不影响生产代码。
