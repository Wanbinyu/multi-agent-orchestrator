# v0.1.0-beta.5 执行清单

**状态**：准备开始；当前第一任务为 B5.1

**目标**：用公开、可复现的数据说明何时单模型更合适，何时多模型协作能提高完成率或降低成本，再把结论固化为有界、可解释、可回退的路由策略。

**发布基线**：`v0.1.0-beta.4`

## 0. 执行原则

- 先建立测量合同，再修改路由行为；不能用路由实现反向定义成功指标。
- 默认基准必须离线、脱敏、零 Provider 调用；真实付费对比需要所有者单独确认模型、次数和费用上限。
- 单模型与 MAO 使用同一任务输入、工作区快照、验证门和完成判定。
- 公开数据不得包含公开编码基准原题、私有项目、密钥或不可再分发内容。
- 每次只推进一个切片；失败结果和“多模型没有优势”的结果同样保留。

## 1. B5.1 可复现基准合同与离线 harness

- [ ] 定义任务 Schema：类别、输入、夹具、允许修改、验证命令、成功条件、风险和来源。
- [ ] 覆盖问答、诊断、小修改、构建、审查和迁移六类最小任务。
- [ ] 定义结果 Schema：输入/输出 token、估算成本、工具调用数、耗时、完成率、误修改率和验证通过率。
- [ ] 单模型与 MAO 共用同一 runner、隔离工作区和确定性验收器。
- [ ] 默认 fixture Provider 使 CI 零费用运行，并验证重复执行不会污染下一轮结果。
- [ ] 输出机器可读 JSON 和简洁 Markdown 报告，保留任务、运行和证据 provenance。

### B5.1 完成门

- 同一夹具连续运行至少三次，成功判定与指标字段稳定。
- 失败、超时、无输出和越界修改均进入失败结果，不被平均值掩盖。
- harness 不读取真实 Key，不调用付费 Provider，不依赖开发机绝对路径。
- Windows/Ubuntu、Python 3.11/3.12 CI 通过。

## 2. B5.2 执行深度合同

- [ ] 定义 `fast`、`standard`、`deep` 的工具、Worker、Reviewer、上下文和验证预算。
- [ ] 简单任务默认不启动 Worker；高风险任务不能用 `fast` 绕过确定性验证。
- [ ] 用户显式选择优先于自动建议，实际深度与原因写入 RunJournal。

## 3. B5.3 可解释模型路由

- [ ] 路由输入只使用任务类型、能力真值、价格、上下文、健康状态和用户约束。
- [ ] 未验证能力不得作为自动升级依据；价格未知时不得宣称节省成本。
- [ ] 路由失败回退到用户指定模型，并限制重试和升级次数。
- [ ] CLI/Web 显示简洁原因，完整决策进入 RunJournal。

## 4. B5.4 单模型与多模型对比

- [ ] 在同一 harness 上运行固定单模型、自动路由和多模型协作三种策略。
- [ ] 至少产出一个公开 token/成本优势案例。
- [ ] 至少产出一个多模型完成率优势案例，或明确记录不应使用多模型的任务类型。
- [ ] 报告区分模型能力、执行深度和协作策略，避免把不同变量混为一个结论。

## 5. B5.5 实验能力

- [ ] 对抗式测试 Worker 仅在实验档启用，尝试推翻实现结果并记录证据。
- [ ] 本地/Ollama 模型可作为零边际成本候选，但健康检查和能力门不降低。
- [ ] 扩充模型目录时继续保持官方来源、`unverified` 回退和 CLI/Web 单一真值。

## 6. B5.6 发布收口

- [ ] 全量测试、安全扫描、分发验收和干净安装通过。
- [ ] 基准任务来源、生成方式、运行命令和结果可公开复现。
- [ ] CHANGELOG、Release Notes、版本号和升级说明完成。
- [ ] Tag 和 GitHub pre-release 仍需所有者单独确认。

## 7. 当前下一步

执行 B5.1，只建立基准合同、脱敏夹具、离线 runner 和结果报告。B5.1 完成前不开始自动路由实现，也不调用真实付费 Provider。
