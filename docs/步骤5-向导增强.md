# 步骤 5：setup 向导增强

## 目标

让 setup 向导在首次运行时生成完整的 `config/providers.yaml`，而不是假设文件已存在。  
新用户只需运行 `python run.py setup`，选择要接入的模型服务并填写 API Key，即可生成所有配置文件。

## 关键实现说明

### 1. Provider 预设

`src/cli/provider_presets.py` 定义了 8 种预设：

| key | 名称 | 协议类型 | 默认 base_url |
|-----|------|----------|---------------|
| `anthropic` | Anthropic (Claude) | anthropic | `https://api.anthropic.com` |
| `openai` | OpenAI | openai | `https://api.openai.com/v1` |
| `glm` | 智谱 GLM | openai | `https://open.bigmodel.cn/api/paas/v4` |
| `deepseek` | DeepSeek | openai | `https://api.deepseek.com/v1` |
| `ark` | 火山方舟 Coding Plan | anthropic | `https://ark.cn-beijing.volces.com/api/coding` |
| `kimi` | Kimi 转发服务 | anthropic | `https://api.va11.icu/` |
| `custom_anthropic` | 自定义 Anthropic 兼容 | anthropic | 用户填写 |
| `custom_openai` | 自定义 OpenAI 兼容 | openai | 用户填写 |

每个预设包含：

- `name`：显示名
- `type`：协议类型（anthropic / openai）
- `base_url`：默认 endpoint
- `env_var`：API Key 对应的环境变量名
- `models`：该 provider 下的默认模型列表及价格
- `model_map`（可选）：逻辑模型名 → 上游真实模型名映射

### 2. 生成 providers.yaml

`build_providers_yaml(selected)` 根据用户选择生成完整 YAML 结构：

```yaml
providers:
  anthropic:
    name: Anthropic (Claude)
    type: anthropic
    base_url: https://api.anthropic.com
    api_keys:
      - ${ANTHROPIC_API_KEY}
    timeout: 120
    rpm_limit: 60
  ...

models:
  claude-sonnet-5:
    provider: anthropic
    model_id: claude-sonnet-5-20251001
    input_price_per_1m: 3.0
    output_price_per_1m: 15.0
  ...
```

### 3. 向导新流程

```
1. 欢迎语
2. 检测并备份已有配置
3. 选择要接入的 Provider（多选）
4. 为每个 Provider 填写 API Key 和可选 base_url
5. 保存 providers.yaml 和 .env
6. 选择使用场景
7. 选择总指挥模型
8. 配置子工程师
9. 保存 workers.yaml
```

### 4. 模型可用性适配

场景预设中的 Worker 默认模型（如 `claude-sonnet-5`）可能不在用户配置的 Provider 中。`adapt_scenario_workers()` 会自动把不可用的默认模型替换为第一个可用模型，并打印提示：

```text
[yellow]Worker '前端开发' 的默认模型 'claude-sonnet-5' 不可用，已替换为 'glm-ark'[/yellow]
```

### 5. 自定义 Provider

对于 `custom_anthropic` / `custom_openai`：

- 必须填写 `base_url`
- 必须至少定义一个模型（显示名 + 上游 model_id + 价格）
- 支持连续添加多个模型

## 遇到的问题与解决方案

### 问题 1：旧向导假设 providers.yaml 已存在

**现象**：原流程先选场景、再选模型，模型列表来自 `get_available_models()`，如果 `providers.yaml` 不存在则返回硬编码的 `DEFAULT_MODELS`，但运行时会因找不到对应 Provider 而失败。

**解决**：把 Provider 配置放到向导第一步，先生成 `providers.yaml`，再用其中的模型列表驱动后续选择。

### 问题 2：场景预设模型与用户配置不匹配

**现象**：软件开发场景的 Worker 默认模型是 `claude-sonnet-5`，但用户只配置了 GLM/DeepSeek，导致模型不可用。

**解决**：增加 `adapt_scenario_workers()`，自动把场景预设中不可用的模型替换为当前可用的第一个模型，并给出提示。

### 问题 3：kimi-for-coding 触发分类器拦截

**现象**：在编辑 `src/cli/setup_wizard.py` 删除旧代码时，工具层提示 `kimi-for-coding is temporarily unavailable`，导致 Edit 操作被拦截。

**解决**：本次未强行删除旧代码中的 `ask_api_keys` / `collect_used_providers` 函数（它们不再被调用），避免触发拦截。后续可在分类器恢复后再清理死代码。

### 问题 4：测试 keyword argument 名称不一致

**现象**：`test_build_provider_config_with_base_url_override` 使用 `base_url_override=` 但函数签名是 `base_url=`。

**解决**：修正测试用例参数名。
