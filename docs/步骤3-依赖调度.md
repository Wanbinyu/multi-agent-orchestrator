# 步骤 3：任务依赖图调度

## 目标

让 Dispatcher 支持按依赖关系（DAG）执行任务。  
之前的 Dispatcher 把所有任务当成无依赖并行执行，无法表达真实软件开发中的先后顺序，例如"先写后端 API，再写前端页面，最后写测试"。

## 关键实现说明

### 1. Task 增加 depends_on

```python
class Task(BaseModel):
    ...
    depends_on: list[str] = Field(default_factory=list, description="依赖的任务 id 列表")
```

默认空数组，表示无依赖，可立即执行。

### 2. Orchestrator 系统提示词更新

在 `config/workers.yaml` 的 Orchestrator 提示词中增加：

```json
{
  "id": "t1",
  ...
  "depends_on": []
}
```

以及依赖规则：

- 无先后顺序时，`depends_on` 为空数组 `[]`
- 任务 B 必须在 A 完成后执行时，B 的 `depends_on` 包含 A 的 `id`
- 依赖关系必须是有向无环图（DAG），不能循环依赖
- 尽量让可并行的任务并行，只在真正需要顺序时才添加依赖

### 3. Dispatcher DAG 调度算法

核心思路：维护一个 `completed` 字典记录已完成任务，动态提交"所有依赖都已成功完成"的任务。

```python
def dispatch(self, plan: TaskPlan, output_dir: str = "output") -> list[TaskResult]:
    tasks = {t.id: t for t in plan.tasks}
    remaining = set(tasks.keys())
    completed: dict[str, TaskResult] = {}
    results: list[TaskResult] = []

    def is_ready(task_id: str) -> bool:
        deps = tasks[task_id].depends_on or []
        return all(dep in completed and completed[dep].success for dep in deps)

    with ThreadPoolExecutor(max_workers=...) as executor:
        futures = {}

        # 初始无依赖任务
        for task_id in list(remaining):
            if is_ready(task_id):
                remaining.remove(task_id)
                futures[executor.submit(...)] = task_id

        while futures:
            future = next(as_completed(futures))
            task_id = futures.pop(future)
            result = future.result()
            completed[task_id] = result
            results.append(result)

            if not result.success:
                self._cascade_failure(task_id, tasks, remaining, completed, results)

            # 新就绪任务入队
            for next_id in list(remaining):
                if is_ready(next_id):
                    remaining.remove(next_id)
                    futures[executor.submit(...)] = next_id
```

### 4. 失败级联

当某个任务失败时，`_cascade_failure` 使用 BFS 把所有直接或间接依赖它的任务标记为失败，避免无效执行：

```python
def _cascade_failure(self, failed_id, tasks, remaining, completed, results):
    queue = [failed_id]
    while queue:
        current = queue.pop(0)
        for task_id in list(remaining):
            deps = tasks[task_id].depends_on or []
            if current in deps:
                remaining.remove(task_id)
                result = TaskResult(task=tasks[task_id], success=False, error=f"依赖任务 {current} 失败")
                completed[task_id] = result
                results.append(result)
                queue.append(task_id)
```

### 5. 循环依赖/不可达任务兜底

调度结束时如果 `remaining` 中还有未处理任务，说明存在循环依赖或上游失败导致无法到达。这些任务会被标记为失败并输出原因。

## 遇到的问题与解决方案

### 问题 1：失败级联只处理了直接依赖

**现象**：第一次实现只把直接下游标记为失败，下游的下游仍留在 `remaining` 中无法执行也不报错。

**解决**：使用 BFS 级联，把新标记为失败的任务也加入队列，处理其下游，直到没有新的受影响任务。

### 问题 2：如何验证调度顺序

**现象**：并发执行导致每次测试的调用顺序可能不同，难以断言严格的执行顺序。

**解决**：测试中不断言绝对时间顺序，而是断言依赖约束：

```python
id_order = [call[0][0].id for call in worker.execute.call_args_list]
assert id_order.index("t2") > id_order.index("t1")
```

### 问题 3：Orchestrator 可能生成循环依赖

**现象**：当前没有主动的循环依赖检测，如果 Orchestrator 输出循环依赖，任务会全部卡在 `remaining` 中。

**解决**：调度结束时把 `remaining` 中剩余任务统一标记为失败，并提示"可能存在循环依赖或上游失败"。  
长期建议：在 Orchestrator 输出后、Dispatcher 执行前增加显式的 DAG 校验。
