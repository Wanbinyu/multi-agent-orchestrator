# MAO 产品方向与 Beta 路线图

**状态**：当前执行路线

**基线版本**：`v0.1.0-beta.4`

**更新日期**：2026-07-19

## 1. 长期目标

MAO 的长期目标是成为一个本地运行、可自托管、能够根据任务选择合适模型，并用更少 token 完成可靠工程工作的多模型 Agent。

项目不以“模型数量”和“并发数量”为成功标准，而以以下结果为成功标准：

- 用户可以使用已有的不同模型套餐，不被单一厂商绑定。
- 简单任务不调用昂贵模型，复杂任务才增加协作深度。
- 工具、修改、测试和风险对用户可见。
- 长任务可以恢复，压缩后不丢失核心需求和证据。
- 没有实现证据或验证证据时不宣称完成。

## 2. 目标用户

- 同时使用国产模型、OpenRouter、Claude、OpenAI 或本地模型的开发者。
- 对 token 成本、套餐额度或地区可用性敏感的个人用户。
- 希望在本机保存配置、会话和项目数据的小团队。
- 需要审查模型执行过程，而不只接受最终答案的工程用户。

当前 Beta 不面向需要强隔离沙箱、企业 SSO、集中审计或 SLA 的生产团队。

## 3. 差异化定位

MAO 不与成熟 Coding Agent 比拼全部功能。优先建立四个可验证优势：

1. **多模型任务路由**：按任务、能力、成本、上下文和可用性选择模型。
2. **token 与上下文控制**：预算、压缩、缓存、项目索引和成本报告可观察。
3. **证据驱动完成**：真实工具结果、验证门、需求矩阵和完成审计形成闭环。
4. **有边界协作**：Worker 有依赖、验收标准、路径所有权和定向重试规则。

## 4. 当前基线

`v0.1.0-beta.2` 已具备：

- `pipx` 安装以及 `mao` / `mao web` 全局入口。
- Provider 配置 UI、主模型选择和连接测试。
- CLI/Web 多轮对话、流式输出和权限模式。
- 多模型任务拆分、Worker 调度和 Reviewer 汇总。
- 工具注册表、原生工具双轨、MCP、Hooks 和本地模型。
- 工程任务分类、Evidence、VerificationGate、CompletionAudit 和 RunJournal。
- 模型窗口配置、动态安全预算和最小长会话基准。
- Windows/Ubuntu、Python 3.11/3.12 CI 和安全扫描。

## 6. 用户验证指标

第一阶段不以 Star 数为主要指标。建议跟踪：

- 10 个外部用户中成功安装的人数。
- 成功配置第一个 Provider 的人数。
- 完成第一次真实项目任务的人数。
- 用户第二次主动使用 MAO 的人数。
- 可复现问题数量和修复周期。
- 单模型与多模型的 token、成本和完成率差异。
- 未授权写入、数据损坏和任务假完成次数，目标必须为 0。

数据收集应以用户主动提交的 Issue、脱敏日志和本地基准为主，不默认上传项目内容或密钥。

外部能力验证按“本地可复现合同 → Terminal-Bench 小规模终端任务 → SWE-bench Lite/Verified”推进。进入 B5.4 时必须主动提醒项目所有者开始真实能力测试，并在任何 Provider 调用前确认模型、次数、费用上限和结果公开范围。Aider 排行榜仅补充代码编辑维度，不替代 MAO 的多模型 Agent 综合评测。

## 7. 优先级规则

出现多个方向时按以下顺序决定：

1. 安全、数据完整性和授权边界。
2. 安装、Provider 连接和首次任务成功率。
3. 完成审计、错误恢复和可观察性。
4. token、成本和长任务稳定性。
5. 模型路由和多 Agent 效果。
6. 新工具、插件、桌面端和生态扩张。

任何新功能都必须回答：它是否提高真实任务成功率，是否减少成本或风险，是否可以被测试。

## 8. 外部参考原则

MAO 可以学习 OpenCode、Aider、Cline、Roo Code 等项目的公开设计，但不以复制功能清单为路线。当前 OpenCode 审计见 [`参考项目-OpenCode.md`](参考项目-OpenCode.md)。

借鉴顺序是：公开接口和交互思想 → 独立设计 → 小范围实现 → 测试验证。复制 MIT 代码时仍必须保留许可证和版权声明；没有必要时优先独立实现。
