# MAO 产品方向与 Beta 路线图

**状态**：当前执行路线

**基线版本**：`v0.1.0-beta.2`

**更新日期**：2026-07-16

## 1. 长期目标

MAO 的长期目标是成为一个本地运行、可自托管、能够根据任务选择合适模型，并用更少 token 完成可靠工程工作的多模型 Agent。

项目不以“模型数量”和“并发数量”为成功标准，而以以下结果为成功标准：

- 用户可以使用已有的不同模型套餐，不被单一厂商绑定。
- 简单任务不调用昂贵模型，复杂任务才增加协作深度。
- 工具、修改、测试和风险对用户可见。
- 长任务可以恢复，压缩后不丢失核心需求和证据。
- 没有实现证据或验证证据时不宣称完成。

## 2. 目标用户

- 同时使用国产模型、OpenRouter、Claude、OpenAI 或本地模型的开发者。
- 对 token 成本、套餐额度或地区可用性敏感的个人用户。
- 希望在本机保存配置、会话和项目数据的小团队。
- 需要审查模型执行过程，而不只接受最终答案的工程用户。

当前 Beta 不面向需要强隔离沙箱、企业 SSO、集中审计或 SLA 的生产团队。

## 3. 差异化定位

MAO 不与成熟 Coding Agent 比拼全部功能。优先建立四个可验证优势：

1. **多模型任务路由**：按任务、能力、成本、上下文和可用性选择模型。
2. **token 与上下文控制**：预算、压缩、缓存、项目索引和成本报告可观察。
3. **证据驱动完成**：真实工具结果、验证门、需求矩阵和完成审计形成闭环。
4. **有边界协作**：Worker 有依赖、验收标准、路径所有权和定向重试规则。

## 4. 当前基线

`v0.1.0-beta.2` 已具备：

- `pipx` 安装以及 `mao` / `mao web` 全局入口。
- Provider 配置 UI、主模型选择和连接测试。
- CLI/Web 多轮对话、流式输出和权限模式。
- 多模型任务拆分、Worker 调度和 Reviewer 汇总。
- 工具注册表、原生工具双轨、MCP、Hooks 和本地模型。
- 工程任务分类、Evidence、VerificationGate、CompletionAudit 和 RunJournal。
- 模型窗口配置、动态安全预算和最小长会话基准。
- Windows/Ubuntu、Python 3.11/3.12 CI 和安全扫描。

## 5. 近期里程碑

### Milestone A：`v0.1.0-beta.3` 用户验证与稳定性

目标：让新用户可以稳定完成第一次真实项目任务。

P0：

- 建立首次安装、首次 Provider 配置、首次项目分析的验收脚本。
- 对至少 3 种真实 Provider 执行人工烟雾测试并记录能力差异。
- 收口 401、429、超时、断网、流式中断和故障切换错误提示。
- 验证 Session、RunJournal 和配置在异常退出后不会损坏。
- 为密钥误提交、工作区越界和未经授权写入增加发布前检查。

P1：

- 建立 Provider / Model 能力矩阵：协议、工具、流式、视觉、上下文来源和已验证日期。
- 改善 `pipx` 未安装、PATH 未刷新和升级失败时的提示。
- 在 Issue 模板中收集平台、模型别名、Provider、任务类型和可复现步骤。

发布门：

- 全量自动测试通过。
- 干净 Windows 与 Linux 环境安装通过。
- 真实 Provider 的分析、修改和失败路径有记录。
- 没有 P0 数据损坏、越权写入或密钥泄露问题。

### Milestone B：`v0.1.0-beta.4` 透明度与长任务

目标：用户无需询问模型，就能理解任务为什么完成或阻塞。

- CLI/Web 展开 WorkPlan、Evidence、VerificationGate、RequirementCheck 和剩余风险。
- 支持会话恢复后继续未完成计划。
- 实现分层压缩和摘要质量门。
- 持久化项目树、文件摘要和内容哈希索引。
- 显示上下文占用、压缩次数、预算来源和估算误差。
- 扩展 32K/64K/128K/200K 多次压缩离线基准。

### Milestone C：`v0.2.0` 路由与基准

目标：用数据证明多模型编排带来的收益。

- 建立真实工程任务基准集。
- 对比单模型与 MAO 的 token、成本、耗时、完成率和误修改率。
- 提供 `fast`、`standard`、`deep` 三档执行深度。
- 路由策略综合任务类型、模型能力、价格、上下文和健康状态。
- 简单任务默认不触发多模型协作；复杂任务只把必要上下文交给 Worker。

发布门：至少形成一个可复现的 token 或完成率优势案例，不用主观宣传代替数据。

### Milestone D：生态与分发

目标：降低安装和扩展门槛，但不早于核心稳定性。

- 更直接的 Windows/macOS/Linux 安装与升级方式。
- Provider、Worker、工具和 MCP 配置 UI。
- 可复用 Agent / Worker 模板。
- 插件清单和版本兼容约束。
- 英文 README、演示素材和贡献者文档。

桌面应用、IDE 扩展和团队服务只有在真实用户需求明确后再立项。

## 6. 用户验证指标

第一阶段不以 Star 数为主要指标。建议跟踪：

- 10 个外部用户中成功安装的人数。
- 成功配置第一个 Provider 的人数。
- 完成第一次真实项目任务的人数。
- 用户第二次主动使用 MAO 的人数。
- 可复现问题数量和修复周期。
- 单模型与多模型的 token、成本和完成率差异。
- 未授权写入、数据损坏和任务假完成次数，目标必须为 0。

数据收集应以用户主动提交的 Issue、脱敏日志和本地基准为主，不默认上传项目内容或密钥。

## 7. 优先级规则

出现多个方向时按以下顺序决定：

1. 安全、数据完整性和授权边界。
2. 安装、Provider 连接和首次任务成功率。
3. 完成审计、错误恢复和可观察性。
4. token、成本和长任务稳定性。
5. 模型路由和多 Agent 效果。
6. 新工具、插件、桌面端和生态扩张。

任何新功能都必须回答：它是否提高真实任务成功率，是否减少成本或风险，是否可以被测试。

## 8. 外部参考原则

MAO 可以学习 OpenCode、Aider、Cline、Roo Code 等项目的公开设计，但不以复制功能清单为路线。当前 OpenCode 审计见 [`参考项目-OpenCode.md`](参考项目-OpenCode.md)。

借鉴顺序是：公开接口和交互思想 → 独立设计 → 小范围实现 → 测试验证。复制 MIT 代码时仍必须保留许可证和版权声明；没有必要时优先独立实现。
