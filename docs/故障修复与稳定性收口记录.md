## 二、故障与根因

### 1. 流式回答重复

**现象**：流式输出一部分后连接中断，从头重试造成正文重复。

**根因**：`chat_stream()` 不区分首个 chunk 前失败和已经向用户输出后的失败。

**处理**：只有零 chunk 时允许重试；任何 chunk 已发出后立即终止并报告中断。

### 2. 429 后 fallback 不完整

**现象**：主模型配额耗尽后只能切到第一个备用模型，备用模型继续失败时无法到达第三级。

**根因**：只读取起始模型的直接 `fallback_models`，未递归读取备用模型自己的 fallback。

**处理**：递归展开并去环；冷却期模型自动跳过且发送通知。

### 3. 错误分类误伤

**现象**：`invalid max_tokens`、错误工具 Schema 等请求错误也会切换模型并进入冷却。

**根因**：用宽泛的 `invalid` / `bad request` 文本判断模型不可用。

**处理**：认证、请求参数、模型不存在/不支持、429、连接错误分别分类；请求参数错误立即暴露，不切换、不标记不健康。

### 4. 配额冷却过短

**现象**：错误明确提示 5 小时额度窗口，但 5 分钟后又会尝试主模型。

**处理**：优先读取 `Retry-After`，否则解析错误中的 hours/minutes/seconds；无法解析时最低 300 秒。

### 5. `/test-models` 诊断不可信

**现象**：命令声称失败模型会被跳过，实际未更新健康状态；Coding Plan 可能聊天成功但诊断失败；注释声称不耗额度。

**根因**：诊断绕开 Provider 正式客户端创建逻辑，并且结果没有写入健康表。

**处理**：通过 Provider 正式 `chat()` 路径发送 `max_tokens=1` 最小请求；失败写入冷却、成功清除冷却；CLI 明示会产生少量 token。

### 6. Windows 目录探查失败

**现象**：模型反复尝试 `dir` / `ls`，被命令白名单拒绝；绝对路径放进 glob pattern 时出现 `Non-relative patterns are unsupported`。

**处理**：新增 `list_dir(path)`；`glob_files(pattern, path)` 将搜索模式和绝对根目录分离；Agent 提示禁止用 shell 探查目录。

### 7. 项目任务漏判或协作后无法继续

**现象**：复杂项目有时走单模型；进入协作后 Worker 即使调用读取工具，也不会拿结果继续生成。

**根因**：协作判断完全依赖一次 LLM JSON；Worker 只有一次模型调用，没有工具循环。

**处理**：明确项目关键字先行判断；Worker 增加最多 5 轮“模型 -> 工具 -> 结果回填 -> 模型”循环，并为旧读取权限补齐目录工具。

### 8. 产生无意义 `generated_N` 文件

**现象**：模型正文代码块被自动抽取为 `generated_1.html` 等，文件名无语义，且可能把示例代码误当成项目产物。

**处理**：Agent / Worker / Reviewer 都不再自动抽取正文代码块。需要项目文件时必须显式 `write_file`；违规时先纠正一次，仍失败则保留 `content.txt` 并标记任务失败。

### 9. Worker 工具权限与协议冲突

**现象**：Worker 能在 Markdown 中调用未配置工具；原生 `tools=` 与“必须 Markdown 工具块”提示同时存在。

**处理**：执行前校验 Worker 工具白名单；原生模式只传结构化 Schema，不再注入 Markdown 工具示例。

### 10. CLI 完成项目分析后不显示最终方案

**现象**：分析 `G:\MAO_test` 时读取数十个文件，控制台最终只显示 `Read(...)`、`Update(未知文件) 0 行`、`response.md` 和 token；Java 重构方案已经生成但用户看不到。

**根因**：CLI 检测到工具块后切换到 spinner，后续 delta 仍被判定包含工具调用；收到 `done` 后只有协作分支会重新打印最终答案。

**处理**：

- `ChatStreamEvent` 增加 `tool_start` / `tool_complete` 结构化过程事件。
- 单 Agent 和协作只要使用过工具，都在 `done` 后渲染最终 `assistant_message`。
- 工具按阶段展示，每阶段展开前 4 项；结束后汇总唯一目录/文件、重复操作、成功/失败和交付文件。
- Agent 提示明确要求工具结束后给出完整结论，禁止以工具调用作为最后输出。

### 11. WebUI 桌面主区过窄与移动端裁切

**现象**：1280px 桌面下聊天中心区只有约 572px；配置页左栏和页面双重滚动。390px 手机下标题、导航、模型表和聊天输入区被裁切，上下文面板内容溢出。

**根因**：固定三栏宽度、高页头、`height:auto` 的手机聊天布局、无响应式容器的模型表，以及 `body` 隐藏横向溢出。

**处理**：

- 页头和导航总高度压缩到约 112px；桌面上下文默认收起，主对话区扩大到约 964px。
- 手机聊天使用 `100dvh`、横向会话条和上下文抽屉，输入区保持在当前视口。
- 配置页使用单一页面滚动，移动端 Provider 横向选择，模型映射转换为纵向卡片。
- 元数据强制在抽屉内换行；新建会话改为应用内对话框。
- 静态资源增加版本参数，解决浏览器继续使用旧 CSS/JS 的问题。

## 五、剩余限制

1. 模型健康状态仅保存在当前进程内，重启后清空。
2. 一次性非流式任务能自动 fallback，但不像流式 CLI/Web 那样实时展示切换卡片；可从响应模型和日志判断。
3. Worker 工具循环上限为 5，超限时要求模型直接收口；极大型项目仍应拆成更多子任务。
4. 协作采用前置整体批准，Worker 内部写文件不再逐个弹窗，这是当前权限设计。
5. 命令执行仍是白名单机制，不是容器级沙箱。
6. 大型项目仍可能在同一轮重复读取相同文件；CLI 会折叠并报告重复次数，但尚未通过读取缓存减少实际调用和 token。
