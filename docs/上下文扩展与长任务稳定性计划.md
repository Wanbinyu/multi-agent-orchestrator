# 上下文扩展与长任务稳定性计划

**状态**：规划中

**创建时间**：2026-07-15

**原则**：以上游真实能力为边界，以长任务稳定性、信息保真和成本可控为目标，不用未经验证的窗口数字制造虚假能力。

## 一、当前基线

MAO 已具备第一版上下文生存能力：

- `ModelConfig.max_context_tokens` 支持按模型覆盖上下文预算。
- 未配置模型窗口时，Agent 使用保守的 `32000 tokens` 默认预算。
- 默认在预算的 `75%`，即当前约 `24000 tokens` 时尝试压缩旧消息。
- 压缩会保留 system 和最近消息，用摘要真正替换较早历史。
- token 计数优先使用 `tiktoken`，不可用时按 UTF-8 字节估算。
- CLI `/context` 可以零模型调用显示模型映射、当前估算、预算来源和压缩阈值。

当前限制：

- `ark-code-latest` 等服务端别名可能动态路由模型，MAO 尚不能确认最终解析到的具体模型版本。
- 模型目录没有维护经验证的上下文窗口、最大输出和建议安全余量。
- 32K 是 MAO 当前安全预算，不等同于 GLM-5.2 或其他上游模型的物理上限。
- 压缩策略只有单层摘要，缺少工具输出压缩、摘要合并、质量校验和恢复记录。
- WebUI 尚不能配置或观察上下文预算。

## 二、目标与非目标

### 目标

1. 每个模型使用有来源、可验证的上下文窗口和输出限制。
2. 根据请求、工具输出和预留生成空间动态计算可用输入预算。
3. 长会话接近窗口时分层释放空间，不因一次大工具结果直接失败。
4. 压缩后保留需求、决策、证据、文件位置、未完成事项和风险。
5. CLI/Web 能解释当前预算、占用、压缩历史和剩余空间。
6. 用离线基准证明连续长任务的稳定性和信息保真度。

### 非目标

- 不突破上游模型的硬上下文限制。
- 不把 Provider 的兼容协议类型当成模型身份或窗口依据。
- 不默认把所有模型设置为 128K、200K 或 1M。
- 不为扩大窗口而无限增加 token 成本。
- 不把长期记忆、项目索引和原始对话历史混成一个无限增长的 prompt。

## 三、预算模型

后续把单一 `max_context_tokens` 拆成可解释的预算：

```text
上游上下文窗口
- 预留输出 token
- Provider/工具协议开销
- 安全余量
= MAO 可用输入预算
```

计划增加的配置语义：

| 配置 | 含义 | 默认策略 |
|---|---|---|
| `context_window_tokens` | 上游声明的硬窗口 | 仅使用已验证目录值或用户显式值 |
| `max_output_tokens` | 单次最大输出 | 按模型目录维护 |
| `context_safety_ratio` | Provider 波动与计数误差余量 | 保守预留 5%-10% |
| `compaction_threshold` | 开始整理上下文的比例 | 默认 75%，按任务风险调整 |
| `tool_result_budget` | 单轮工具结果预算 | 超限先压缩工具结果 |

旧的 `max_context_tokens` 保持兼容，并在迁移期解释为 MAO 安全预算。

## 四、实施阶段

### Context 1：模型窗口真值与配置面（P0）

- 为内置模型目录增加窗口、最大输出、信息来源和验证日期。
- 区分固定模型 ID 与 `ark-code-latest` 这类服务端动态别名。
- 对未知模型继续使用 32K 保守预算，并明确显示“未验证”。
- WebUI 增加上下文窗口、安全比例、压缩阈值和输出预留字段。
- 配置保存时校验范围，禁止压缩阈值大于安全输入预算。
- `/context` 增加“上游窗口 / MAO 安全预算 / 当前可用预算”分层展示。

**验收**：任何启用模型都能说明预算来源；未知值不会伪装成官方上限。

### Context 2：动态 ContextBudgetManager（P0）

- 新增统一预算管理器，Agent、Worker、Reviewer 和 Compactor 共用。
- 请求前计算 system、历史、工具 Schema、记忆和预留输出占用。
- 根据任务类型设置输出预留：回答、代码生成、项目审查使用不同额度。
- Provider 支持时使用对应 tokenizer；不支持时保留保守估算和安全余量。
- 超预算时给出结构化原因，不直接依赖上游 400 错误。

**验收**：同一请求在 CLI/Web/Worker 路径得到一致预算；发送前可证明不会超过安全窗口。

### Context 3：分层压缩与信息保真（P1）

按低损失到高损失顺序释放空间：

1. 去除重复工具结果和已经失效的中间输出。
2. 对超长文件、命令和搜索结果生成带来源的局部摘要。
3. 合并旧对话摘要，保留最近完整回合。
4. 极限情况下生成任务检查点，明确丢弃范围和剩余风险。

同时增加：

- 摘要 Schema：需求、决策、事实、证据引用、修改文件、待办、风险。
- 摘要质量门：关键字段缺失时不替换原历史。
- 压缩失败降级：减少工具注入或提示新会话，不中断主流程。
- RunJournal 记录压缩前后 token、策略、耗时和失败原因。

**验收**：多次压缩后仍能复述核心需求、已改文件和未完成事项；不编造不存在的证据。

### Context 4：持久项目上下文（P1）

- 项目树、文件摘要、符号索引按项目根目录和内容哈希复用。
- prompt 中优先放引用和摘要，需要时再按需读取原文。
- 文件变化只失效相关摘要，不重建整个项目上下文。
- 会话检查点与长期记忆分离：前者服务任务恢复，后者保存稳定事实。

**验收**：重复分析同一项目时，读取文件数和输入 token 显著下降，变更文件仍能及时刷新。

### Context 5：CLI/Web 可观察性（P1）

- CLI 状态栏和 Web 工作区显示占用比例、剩余预算和预算来源。
- 压缩发生时显示一次简洁事件，不输出隐藏思维或完整摘要内容。
- 支持查看压缩次数、最近压缩时间、压缩前后 token 和保留消息数。
- 预算异常、动态模型别名和估算误差使用明确警告，而不是静默猜测。

**验收**：用户无需询问模型即可理解当前上下文状态和压缩行为。

### Context 6：长任务基准与发布门（P0）

- 建立 32K、64K、128K、200K 的离线模拟窗口测试，不在 CI 调用付费模型。
- 场景覆盖长对话、大工具输出、项目重构、多次压缩、会话恢复和模型切换。
- 指标包括完成率、关键信息保留率、压缩次数、输入 token、延迟和成本。
- 真实模型只做人工烟雾测试，并记录模型、端点、窗口来源和费用。

**发布门**：

- 所有预算路径在超限前可预测地压缩或阻止请求。
- 连续三次压缩后关键需求与文件证据保留率达到基准要求。
- 未知模型窗口不会被展示为已确认数值。
- 失败不会损坏 Session、RunJournal 或项目索引。

## 五、实施顺序

建议顺序：

1. Phase 7.1 任务分类与 Context 1 同步推进，为不同任务建立预算依据。
2. 在 Phase 7.2 证据循环前完成 Context 2，避免项目侦察再次无界消耗上下文。
3. Context 3 与 Phase 7.3 验证门结合，压缩结果也必须有质量检查。
4. Context 4/5 在核心预算稳定后实施。
5. Phase 7.4 完成后进入开源验收；公开 Beta 前至少完成 Context 1-2 和 Context 6 的最小长会话基准。
6. Context 3-5 可在 Beta 后继续，但扩大默认窗口或宣传高级长任务能力前必须通过完整 Context 6。

## 六、风险与控制

| 风险 | 控制 |
|---|---|
| 官方窗口或动态别名变化 | 保存来源与验证日期；未知时回退保守预算 |
| token 估算偏低 | Provider tokenizer + 安全余量 + 发送前复核 |
| 摘要丢失关键事实 | 结构化摘要、证据引用、质量门和检查点 |
| 压缩本身消耗过多 | 优先本地裁剪/去重，限制摘要输出和调用频率 |
| 大窗口导致成本上升 | 按任务分配预算，默认不填满窗口 |
| UI 配错窗口 | 合理范围校验、危险值警告和恢复默认值 |

## 七、近期下一步

- 不立即修改当前 32K 默认值。
- 先确认火山方舟 Coding Plan 当前所选 GLM-5.2 的官方窗口、最大输出和动态别名语义。
- 完成 Context 1 的模型目录字段与配置校验设计。
- 为 32K 默认预算建立长会话离线基准，作为后续扩展到更大窗口的对照组。
