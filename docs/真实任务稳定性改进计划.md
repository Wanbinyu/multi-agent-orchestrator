## 1. 已确定的标准操作方式

### 1.1 用户操作流程

复杂工程任务优先使用持久化 Plan 模式：

```text
mao
/plan enter 在 G:\目标目录 创建项目，并说明技术栈、范围和验收标准
# 发送完整需求，等待只读侦察与多模型方案
/plan show
/plan revise <修订意见>   # 可选
/plan approve             # 批准后才进入实现链
/runs                     # 查看证据、验证门和阻塞原因
/context                  # 查看 token 预算和压缩状态
```

WebUI 使用顶部 `Plan` 控件，流程与 CLI 相同：进入 → 生成方案 → 修订或批准 → 正常执行。原 `/plan <需求>` 保留为一次性协作入口，但大型项目默认推荐持久化 Plan。

### 1.2 规则与权限

- 项目规则：目标项目根或子目录放置 `AGENTS.md`、`CLAUDE.md` 或 `.mao/rules/*.md`。
- 用户权限：`config/permissions.yaml`；项目权限：`<project>/.mao/permissions.yaml`。
- 权限优先级：硬只读/Plan 边界 → `deny` → `ask` → `allow` → 会话模式默认。
- `auto` 自动执行允许的工具；`approve` 自动读取、非只读工具询问；`readonly` 拒绝所有非只读工具。
- 主 Agent、Orchestrator、Worker、Reviewer 接收同一项目规则；主 Agent 和 Worker 在执行边界共用权限引擎。

### 1.3 多模型工程流程

```text
只读侦察
  → Planning Council（证据检查 / 架构 / 批评 / 综合）
  → 用户批准
  → Orchestrator 拆分任务与所有权
  → Worker 实现与验证
  → Reviewer 对照需求和证据审查
  → CompletionAudit 决定 completed 或 blocked
```

任何模型文字都不能代替真实工具证据。修改任务至少需要针对性验证和相邻回归；高风险新项目还需要集成、全量、运行时 smoke 和使用说明。验证不完整时必须保持 `blocked`。

## 3. 根因边界

### 3.1 MAO 必须修复

1. 自然语言构建意图覆盖不足：`给我做一个`、`现在给我做` 等句中命令未进入 build。
2. 任务分类与实际行为脱节：实际产生项目写入后没有动态提升风险和验证深度。
3. 大型构建误走单 Agent，Orchestrator/Worker/Reviewer 没有被调用。
4. 命令工具缺少稳定的结构化 `cwd`、跨平台命令模板和脚本发现。
5. 前端完成审计只看文件/命令证据，没有浏览器运行时错误、路由和数据渲染门。
6. 会话总结只依赖当前上下文，无法可靠聚合所有 RunJournal 和修复历史。

### 3.2 生成项目模板/模型必须改进

1. Mock 层不能把响应对象作为 Axios 请求配置返回；需要采用自定义 adapter、mock library 或独立 service 实现。
2. 路由引用必须在完成前检查目标文件闭包。
3. `package.json` 脚本、devDependencies 和配置文件必须一致。
4. 大屏、390px 窄屏、登录和全部主导航路由必须进入 smoke 清单。
5. 新项目默认建立 Git 基线，除非用户明确禁止；提交仍需单独授权。
