# 多 Agent 编排工具迭代计划

> 基于 `docs/开发记录.md`、`docs/ccswitch学习笔记.md` 以及对当前代码的梳理，制定本迭代计划。  
> 原则：**一步一步来，完成一步写好 md 后再写下一步。**

> 状态：**全部 5 个步骤已完成**（2026-07-10）  
> 当前测试覆盖：31 个单元测试全部通过

---

## 当前状态

CLI MVP 已能跑通基本流程：

```
用户需求 → Orchestrator 拆任务 → Dispatcher 并发调度 Worker → 代码块写入 output → summary.md
```

已支持：多 Provider 接入、多 key 轮询、指数退避重试、Token 计费、配置向导。

---

## 主要缺口

1. **Reviewer 悬空**：`config/workers.yaml` 中配置了 `reviewer`，但 `run.py` 执行完 Worker 后直接汇总，没有审查/整合环节。
2. **无任务依赖调度**：`dispatcher.py` 把所有任务并行执行，无法表达"先写后端 API，再写前端，再写测试"的依赖关系。
3. **无模型降级/备用路由**：某模型失败只能重试同一模型，不能切到备用模型。
4. **无 Provider 层模型映射**：CCswitch 式转发服务通常把所有 Claude 模型名映射到同一个上游模型，当前代码直接发送逻辑模型名 upstream。
5. **工具能力弱**：只有 `write_file`，Worker 无法读文件、执行命令、运行测试。
6. **setup 向导不写 `providers.yaml`**：新用户仍需手动配置 Provider。

## 迭代步骤

### 步骤 1：Reviewer 收口流程

**目标**：让 Orchestrator → Workers → Reviewer → 最终输出 形成闭环。

**要修改的文件**：
- `src/core/reviewer.py`（新增）
- `run.py`
- `config/workers.yaml`
- `src/models/schemas.py`（如需要）

**预期行为**：
- Worker 全部执行完后，把所有子任务结果和原始需求一起交给 `reviewer` 模型。
- Reviewer 输出 `ReviewResult`：是否通过、问题列表、`final_output`。
- 把 `final_output` 追加到 `summary.md`，并输出到终端。
- 如果未通过，可输出改进建议（本次不实现自动重试，仅记录）。

**验收标准**：
- 运行 `python run.py "开发一个前后端登录功能"` 后，终端能看到 Reviewer 的审查结论。
- `summary.md` 中多出 Reviewer 的 `final_output` 段落。

---

### 步骤 2：Provider 层模型映射

**目标**：解决 CCswitch 式转发服务的模型名映射问题，让逻辑模型名和上游真实模型名解耦。

**要修改的文件**：
- `src/models/schemas.py`
- `src/gateway/provider.py`
- `src/gateway/router.py`
- `config/providers.yaml`

**预期行为**：
- 在 Provider 配置中增加可选字段：
  - `api_format`: `anthropic`（默认） / `openai_chat`
  - `model_map`: 模型别名 → 上游真实模型名
  - `auth_strategy`: `x-api-key` / `bearer`（未来扩展）
- GatewayClient 在请求前，先把 `model_id` 通过 `model_map` 转换。
- 保留当前 `anthropic` / `openai` 类型行为作为默认，不破坏现有配置。

**验收标准**：
- 配置 `kimi` provider 的 `model_map` 把所有 `claude-*` 映射到同一上游模型后，请求能正常返回。
- 新增单元测试验证 `model_map` 转换逻辑。

---

### 步骤 3：任务依赖图调度

**目标**：支持带依赖关系的子任务，让 Dispatcher 能按 DAG 执行。

**要修改的文件**：
- `src/models/schemas.py`（Task 增加 `depends_on`）
- `src/core/orchestrator.py`（生成任务时输出依赖）
- `src/core/dispatcher.py`（DAG 调度）

**预期行为**：
- `Task` 增加 `depends_on: list[str]` 字段。
- Orchestrator 在拆任务时，可以表达任务之间的依赖。
- Dispatcher 使用拓扑排序或按依赖状态轮询，先执行无依赖任务，再执行依赖已完成的任务。
- 失败任务阻塞下游任务执行，并记录原因。

**验收标准**：
- 运行一个带依赖的需求（如"先写后端 API，再写前端页面"），任务按正确顺序执行。
- 测试用例覆盖：无依赖并行、有依赖串行、依赖失败下游阻塞。

---

### 步骤 4：扩展 Worker 工具集

**目标**：让 Worker 能读取已有文件、执行受控的命令/测试。

**要修改的文件**：
- `src/tools/file_tools.py` 重命名为 `src/tools/worker_tools.py` 或新增 `src/tools/read_file.py`、`src/tools/run_command.py`
- `src/core/worker.py`
- `src/models/schemas.py`

**预期行为**：
- 新增 `read_file` 工具：读取指定文件内容返回给模型。
- 新增 `run_command` 工具：执行白名单内的命令（如 `python -m pytest`、`npm test`），需要用户确认或配置允许列表。
- Worker 系统提示词中说明可用工具及调用格式。
- Orchestrator 可在任务中指定需要使用的工具。

**验收标准**：
- Worker 能读取 `output/` 下已生成的文件作为上下文。
- Worker 能运行 `pytest` 并获取结果。
- 危险命令默认被拒绝。

---

### 步骤 5：setup 向导生成 providers.yaml

**目标**：降低新用户配置门槛，向导结束时自动生成完整的 `config/providers.yaml`。

**要修改的文件**：
- `src/cli/setup_wizard.py`
- `config/providers.yaml`（作为模板参考）

**预期行为**：
- 向导除了询问总指挥/子工程师外，还询问：
  - 有哪些 Provider（Anthropic / OpenAI / GLM / DeepSeek / 火山方舟 / Kimi 转发 / 自定义）
  - 每个 Provider 的 base_url、api_key、默认模型
  - 是否启用模型映射
- 根据输入生成 `config/providers.yaml` 和 `.env`。
- 支持"从 CCswitch JSON 导入"作为高级选项（可选）。

**验收标准**：
- 删除 `config/providers.yaml` 后运行 `python run.py setup`，能重新生成可用配置。
- 生成的配置能直接运行 `python run.py "需求"`。

## 建议执行顺序

1. **步骤 1（Reviewer）**：改动最小、对输出质量提升最明显，且与项目原始"验收结果"定位一致。
2. **步骤 2（模型映射）**：解决当前实际接入转发服务时的模型名问题，是 CCswitch 学习后直接可落地的改进。
3. **步骤 3（依赖调度）**：让编排器能处理真实软件开发的流水线。
4. **步骤 4（扩展工具）**：让 Worker 具备读写、执行能力，才能生成可运行项目。
5. **步骤 5（向导增强）**：把配置流程彻底闭环，提升首次使用体验。
