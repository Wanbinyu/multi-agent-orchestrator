# 步骤 4：扩展 Worker 工具集

## 目标

为 Worker 增加 `read_file` 和受控的 `run_command` 工具，让 Worker 能：

- 读取项目内已有文件的内容（如已有代码、配置）
- 运行测试、构建命令等白名单内的命令

从而从"只写代码"升级为"能读写、能执行"的完整执行单元。

## 关键实现说明

### 1. 工具调用格式

模型在回复中使用 Markdown 代码块调用工具：

````markdown
```tool:read_file
{"path": "backend_t1/generated_1.py"}
```

```tool:run_command
{"command": "pytest"}
```
````

Worker 执行完工具后，把结果替换回原文，再写入文件。

### 2. read_file 工具

```python
def read_file(path: str, base_dir: str = ".") -> ToolResult:
    target = _resolve_path(path, base_dir)
    with open(target, "r", encoding="utf-8") as f:
        content = f.read()
    return ToolResult(success=True, output=content)
```

安全控制：

- 使用 `Path.resolve()` + `relative_to()` 防止目录穿越（`../etc/passwd` 会被拒绝）
- 只能读取 `base_dir` 下的文件

### 3. run_command 工具

```python
def run_command(
    command: str,
    base_dir: str = ".",
    allowed_prefixes: list[str] | None = None,
    timeout: int = 60,
) -> ToolResult:
    if not _is_command_allowed(command, allowed_prefixes):
        return ToolResult(success=False, error="命令不在白名单内...")
    args = shlex.split(command)
    result = subprocess.run(args, cwd=base_dir, capture_output=True, text=True, timeout=timeout, shell=False)
    ...
```

安全控制：

- **白名单前缀匹配**：只有命令以允许前缀开头才会执行
- **默认白名单**：`pytest`、`python -m pytest`、`python `、`npm `、`node `、`npx `、`git status/diff/log`
- **禁用 shell**：使用 `shlex.split` + `subprocess.run(..., shell=False)`，避免 shell 注入
- **超时控制**：默认 60 秒
- **目录限制**：在 `base_dir`（即当前 Worker 的输出目录）下执行
- **自定义白名单**：可在 `workers.yaml` 的 Worker 配置中通过 `allowed_commands` 覆盖

### 4. Worker 中的工具集成

`src/core/worker.py` 中：

1. 根据 Worker 的 `tools` 列表生成工具说明并拼接到用户输入中
2. 调用模型获取回复
3. 使用 `process_tool_calls()` 解析并执行工具调用
4. 把工具执行结果替换到内容中
5. 最后写入代码块到文件

```python
system_prompt = worker_cfg.get("system_prompt", "")
tools = worker_cfg.get("tools", ["write_file"])
user_content = f"""...
{build_tool_instructions(tools)}
请直接输出..."""

response = self.gateway.chat(...)
content, tool_results = process_tool_calls(response.content, task_output_dir, allowed_prefixes)
files_written = write_output_files(content, task_output_dir)
```

## 遇到的问题与解决方案

### 问题 1：工具调用参数 JSON 解析失败时没有记录结果

**现象**：测试 `test_process_tool_calls_invalid_json` 失败，`results` 为空。

**原因**：`process_tool_calls` 的 `replacer` 在 `except json.JSONDecodeError` 分支直接 `return`，没有向 `tool_results` 追加记录。

**解决**：在异常分支也追加一个失败的工具结果：

```python
except json.JSONDecodeError as e:
    tool_results.append({
        "tool": tool_name,
        "params": {},
        "success": False,
        "output": "",
        "error": f"参数解析失败：{e}",
    })
    return f"\n[工具调用参数解析失败：{e}]\n"
```

### 问题 2：命令注入风险

**现象**：如果模型输出 `python -c "import os; os.system('rm -rf /')"`，直接执行会很危险。

**解决**：

1. 使用 `shlex.split` + `shell=False` 避免 shell 元字符注入
2. 使用白名单前缀控制可执行命令范围
3. 超时限制，防止长时间挂起
4. 工作目录限制在 Worker 输出目录

> 注意：白名单机制不是绝对安全，仍建议只在可信模型和本地环境中使用。

### 问题 3：路径穿越

**现象**：`read_file` 如果允许 `../` 可能读取项目外文件。

**解决**：使用 `Path.resolve()` 并检查 `target.relative_to(base)`，越界时抛出 `ValueError`。
