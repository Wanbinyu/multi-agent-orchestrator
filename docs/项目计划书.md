> 目标：打造一个**总工模型（Fable 5 / Opus）+ 多外部便宜模型并发协作**的 Agent 编排工具。

## 一、项目定位

做一个**本地/私有部署的模型路由与任务编排中枢**：

- **总指挥（Orchestrator）**：固定接 Anthropic Claude Fable 5 或 Opus 4.8，负责需求理解、任务拆解、子 Agent 调度、结果验收。
- **执行队（Workers）**：并发接入 GLM、DeepSeek、Qwen、GPT-4o-mini、Haiku 等便宜/垂直模型，按任务类型动态分配。
- **模型网关（Gateway）**：统一管理各家 API 的 key、endpoint、重试、降级、计费统计，替代 CCswitch「一次只能连一个模型」的限制。

核心口号：**一次需求输入，多模型并发执行，总指挥统一收口。**

---

## 二、为什么要自带模型网关（关于 CCswitch）

你提到 CCswitch 一次只能连接一个模型，这对我们的场景是不够的。因为：

| 场景 | CCswitch 现状 | 本工具需要 |
|------|---------------|-----------|
| 总工模型分析 | 一次切一个 | Fable 5 常驻，不被切换掉 |
| 子 Agent 并发 | 不支持 | 前端、后端、测试同时跑 |
| 任务中途换模型 | 手动切换 | 自动按任务类型路由 |
| 失败自动降级 | 无 | 模型超时/报错后切备用 |
| 多 key 轮询 | 无 | 多账号负载均衡 |

**结论：必须自带模型网关功能**，而不是依赖 CCswitch 做单连接切换。本工具内部要实现一个轻量版、可多并发、可路由的「CCswitch Pro」。

## 四、核心模块设计

### 4.1 用户交互层

| 形态 | 用途 | 优先级 |
|------|------|--------|
| CLI | 开发者日常、脚本集成 | P0 |
| Web UI | 可视化流程、非技术用户 | P1 |
| API Server | 被其他系统调用 | P1 |
| VS Code 插件 | 代码场景深度集成 | P2 |

### 4.2 Orchestrator 总工层

Fable 5 的 Prompt 模板设计：

```yaml
system: |
  你是项目总工程师。用户会给出一个开发需求。
  你必须：
  1. 分析需求并拆成可并行子任务；
  2. 为每个子任务指定：类型、输入、输出格式、验收标准；
  3. 从可用模型列表中选出最合适的执行模型；
  4. 待所有子任务返回后，做一致性检查、接口对齐、最终整合。

available_workers:
  frontend: { model: claude-sonnet-5, desc: "React/Vue 组件开发" }
  backend:  { model: glm-4, desc: "API/数据库/业务逻辑" }
  test:     { model: claude-haiku-4-5, desc: "写单元测试" }
  doc:      { model: glm-4-flash, desc: "生成文档" }
```

输出必须是结构化 JSON，例如：

```json
{
  "tasks": [
    {
      "id": "t1",
      "type": "frontend",
      "title": "登录页面组件",
      "input": "...",
      "output_format": "tsx + css",
      "acceptance": "包含邮箱/密码校验、错误提示",
      "assigned_model": "claude-sonnet-5"
    },
    {
      "id": "t2",
      "type": "backend",
      "title": "登录 API",
      "input": "...",
      "output_format": "python fastapi",
      "acceptance": "返回 JWT、密码 bcrypt 加密",
      "assigned_model": "glm-4"
    }
  ]
}
```

### 4.3 Task Dispatcher 调度器

- 基于有向无环图（DAG）解析任务依赖。
- 使用线程池/协程并发执行无依赖任务。
- 每个 Worker 独立上下文，互不干扰。

### 4.4 Worker 执行层

每个 Worker = 角色 Prompt + 指定模型 + 工具权限。

| Worker | 推荐模型 | 工具权限 |
|--------|---------|---------|
| FrontendDev | Sonnet 5 / GPT-4o | 读写前端文件、运行 npm |
| BackendDev | GLM-4 / DeepSeek-V3 | 读写后端文件、运行 pytest |
| Tester | Haiku 4.5 / GLM-4-Flash | 运行测试、读取代码 |
| Reviewer | Fable 5 / Sonnet 5 | 只读审查、输出报告 |
| DocWriter | 最便宜模型 | 读写 docs/ |

### 4.5 Model Gateway 模型网关（替代 CCswitch 的核心）

必须实现：

1. **Provider 统一接口**：
   - OpenAI 兼容格式（OpenAI、GLM、DeepSeek、Qwen 等多数国内模型都兼容）
   - Anthropic Messages API
   - Azure OpenAI
   - 可扩展其他私有协议

2. **连接池 + 多 key 管理**：
   - 每个 Provider 配置多个 API key，轮询或随机使用。
   - 单 key 限流时自动切下一个。

3. **路由策略**：
   - 按任务类型路由（frontend → sonnet, backend → glm）
   - 按成本路由（预算紧张时选最便宜）
   - 按质量路由（关键任务走 Fable 5）
   - 手动指定

4. **容错机制**：
   - 超时重试（指数退避）
   - 模型失败降级到备用模型
   - 流式输出中断保护

5. **计费统计**：
   - 每次请求记录：模型、token 数、费用、耗时、任务 ID。
   - 提供会话/项目/任务维度的成本报表。

## 五、推荐技术栈

| 层级 | 推荐方案 | 理由 |
|------|---------|------|
| 后端 | Python 3.11+ + FastAPI | 生态成熟、LLM SDK 丰富 |
| 编排 | 自研 Dispatcher + Pydantic | 轻量、可控，不必强绑 LangChain |
| LLM SDK | `openai` + `anthropic` + `httpx` | 直接调用各家原生接口 |
| 数据库 | SQLite / PostgreSQL | 任务状态、日志、计费持久化 |
| 前端 | React + Tailwind | Web UI 监控任务和成本 |
| 配置 | YAML / TOML | 用户配置 Provider、Worker、路由 |
| 部署 | 本地 EXE / Docker / pip 包 | 先本地跑通，再考虑服务化 |

## 九、风险与对策

| 风险 | 对策 |
|------|------|
| 便宜模型输出质量不稳 | Orchestrator 做二次审查，不通过则重试或升级模型 |
| 多模型接口格式差异 | Gateway 层统一封装为 OpenAI 兼容格式 |
| 并发导致 API 限流 | 多 key 轮询 + 请求队列 + 指数退避 |
| 上下文传递不完整 | 每个任务携带独立、完整的输入与验收标准 |
| 工具执行安全风险 | 默认只读，写操作需用户确认或白名单 |

## 十、与 CCswitch 的关系总结

- **不要复用 CCswitch 做底层连接**，因为它的设计是「单连接手动切换」，无法支撑多 Agent 并发。
- **本工具要内置一个增强版网关**：保留 CCswitch 的「便捷切换」优点，但升级为「多并发、自动路由、限流降级、计费统计」。
- 未来如果 CCswitch 开放多并发 API，可以考虑把它的路由策略接入本工具作为可选 Provider，但不是当前必要路径。
