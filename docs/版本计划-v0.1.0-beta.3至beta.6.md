# MAO v0.1.0-beta.3 至 beta.6 版本计划

**状态**：已确定，等待按版本执行

**基线**：`v0.1.0-beta.2`

**制定日期**：2026-07-16

**修订记录**：2026-07-17 —— beta.4 增加上下文干扰度指标、L0/L1/L2 分层参考与 Reviewer 信息限制验证；beta.5 增加预设模型目录扩充、基准防污染、对抗式测试生成与本地模型成本档。

## 1. 版本拆分原则

后续四个 Beta 每个版本只承担一个主要结果，避免把 Provider、上下文、路由和插件同时改动：

| 版本 | 主要结果 | 不在本版范围 |
|---|---|---|
| `beta.3` | Provider/Claude 可信接入与首次使用稳定性 | 大型 UI 改版、插件市场 |
| `beta.4` | 工程透明度、会话恢复与长任务上下文 | 智能路由、Provider 扩张 |
| `beta.5` | 模型路由、执行深度和真实基准 | 桌面应用、IDE 扩展 |
| `beta.6` | 受控 Plugin API v0 与扩展诊断 | 在线插件市场、自动执行未知代码 |

只有当前版本的发布门全部通过后，才进入下一版本。紧急安全修复可以插入补丁版本，但不得借机扩大功能范围。

## 2. `v0.1.0-beta.3`：Provider 与 Claude 稳定性

### 目标

让新用户能够完成“安装 → 配置 Provider → 测试连接 → 执行第一次真实任务”，并使官方 Anthropic Claude 成为经过验证的 Provider，而不是只有预设名称。

### P0 范围

1. **Provider 能力真值**
   - 为模型记录协议、上游模型 ID、流式、原生工具、视觉、reasoning、上下文窗口、输出限制、信息来源和验证日期。
   - 未验证能力保持关闭或标记 `unverified`，不根据模型名称猜测。
   - 动态别名与固定版本模型分开显示。

2. **官方 Anthropic Claude 验证**
   - 审计官方预设中的模型 ID、价格、上下文和能力标签。
   - 验证 `ANTHROPIC_API_KEY`、官方 Messages API、流式响应和错误分类。
   - 验证原生 `tool_use` 的完整回合，不只验证 Schema 生成。
   - 明确 Claude 订阅套餐与 Anthropic API Key 不是同一配置入口，禁止使用非公开登录凭据。

3. **统一 Provider 错误**
   - 对 401/403、404、429、超时、连接失败、5xx、上下文超限和不支持能力生成结构化错误。
   - CLI/Web 显示用户可执行的下一步，不暴露密钥或完整响应头。
   - 只有瞬时错误进入重试或故障切换；认证和配置错误立即停止。

4. **首次使用验收**
   - 干净目录中验证 `mao` 首次向导和 `mao web` 配置页。
   - 干净 Windows 与 Linux 环境验证 `pipx` 安装、升级和卸载。
   - 异常退出后 Provider 配置、Session 和 RunJournal 不损坏。

5. **扩展加载诊断**
   - Hooks/MCP 加载失败不再完全静默。
   - 启动时提供有界错误摘要和 `doctor` 级诊断信息，但不在本版引入完整插件系统。

### P1 范围

- 建立 GLM/Ark Coding、Anthropic Claude、OpenAI 兼容和本地模型的兼容性表。
- Issue 模板收集平台、Provider、逻辑模型名、上游模型 ID、任务类型和脱敏错误。
- 文档增加常见鉴权差异和“套餐不等于 API 额度”的说明。

### 发布门

- 全量自动测试、Python/JavaScript 静态检查和 `git diff --check` 通过。
- Windows/Linux 干净安装通过。
- 至少 3 条 Provider 路径有记录，其中官方 Claude 在没有可用 Key 时至少完成离线契约测试；真实调用必须由所有者显式授权。
- 认证失败、限流、超时和流式中断不会损坏本地状态。
- 没有 P0 密钥泄露、越界写入、数据损坏或任务假完成问题。

详细实施顺序见 [`Beta3-执行清单.md`](Beta3-执行清单.md)。

## 3. `v0.1.0-beta.4`：透明度与长任务

### 目标

用户无需询问模型，就能看懂任务计划、执行证据、验证结果、阻塞原因和上下文行为。

### 范围

1. CLI/Web 可展开查看：
   - `WorkPlan`
   - `Evidence`
   - `VerificationGate`
   - `RequirementCheck`
   - `CompletionAudit`
   - `residual_risks`
2. 会话恢复后识别 `running`、`blocked` 和未完成计划，要求用户确认后继续。
3. 实现 Context 3 分层压缩：去重、局部摘要、旧回合摘要和任务检查点。
4. 为摘要增加需求、决策、证据、修改文件、待办和风险 Schema，以及质量门。
5. 实现项目树、文件摘要和内容哈希索引的增量复用。
6. CLI/Web 显示上下文占用、压缩次数、预算来源、估算误差和最近压缩事件。
7. 压缩质量门增加上下文干扰度指标：压缩后任务相关 token 必须占主导，不只用保留率衡量质量；冗余上下文会拉低模型表现，干扰度需可测量、可回放。
8. 压缩分层参考 L0/L1/L2 抽象（索引/摘要/全文），保留按需展开通道，避免压缩成为单向丢弃。
9. Reviewer 采用信息限制验证：对照需求与证据独立验证，不阅读 Worker 自述；并行独立验证优先于多轮评审循环，验证方式差异写入 RunJournal。

### 发布门

- 连续三次压缩后仍保留核心需求、文件变更和证据引用。
- 压缩后任务相关 token 占比通过约定门槛，干扰度可测量、可回放。
- 会话中断恢复不会自动重复写入或重复执行成功任务。
- UI 默认简洁，展开后信息完整；390px 移动视口无溢出。
- 32K/64K/128K/200K 离线窗口基准通过约定门槛。

## 4. `v0.1.0-beta.5`：路由与可复现基准

### 目标

用公开数据证明何时应该使用单模型，何时多模型协作能够提高完成率或降低成本。

### 范围

1. 建立真实工程任务基准集：问答、诊断、小修改、构建、审查和迁移。
2. 记录单模型与 MAO 的：
   - 输入/输出 token
   - 估算成本
   - 工具调用数
   - 总耗时
   - 完成率
   - 误修改率
   - 验证通过率
3. 提供 `fast`、`standard`、`deep` 三档执行深度。
4. 路由综合任务类型、能力、价格、上下文、健康状态和用户约束。
5. 简单任务默认不触发 Worker；复杂任务只发送必要上下文。
6. 路由决策写入 RunJournal，并给出简洁可解释原因。
7. 扩充预设模型覆盖：`src/models/catalog.py` 作为单一真值源，覆盖 Anthropic、OpenAI、DeepSeek、智谱 GLM、Kimi、阿里 Qwen、MiniMax、字节豆包和 Google Gemini；未核实条目保持 `unverified`，CLI 与 Web 预设统一由目录生成。
8. 基准任务集使用私有或程序化生成任务和可复现 harness，明确规避公开基准污染争议；任务来源与生成方式可审查。
9. 增加对抗式测试生成实验档：独立测试 Worker 试图推翻实现 Worker 的结果，对抗过程与结论写入 RunJournal。
10. 路由成本档纳入本地/开源模型：`fast` 档默认零边际成本模型（Ollama 等），复杂任务才升级到付费模型。

### 发布门

- 至少一个公开、可复现的 token/成本优势案例。
- 至少一个多模型完成率优于单模型的案例，或明确证明某类任务不应使用多模型。
- 基准集不包含公开基准原题，任务来源和生成方式可审查。
- 路由失败可回退到用户指定模型，不会无界重试。
- 基准不包含私有项目、密钥或不可公开数据。

## 5. `v0.1.0-beta.6`：Plugin API v0

### 目标

把当前 ToolSource、MCP、Hooks 和 Provider 预设整理为可诊断、可版本约束、必须显式启用的扩展接口。

### 范围

1. 定义插件清单：名称、版本、MAO API 版本、入口、能力、权限和来源。
2. 使用 Python entry point 或等价标准发现已安装插件，不扫描任意工作区代码。
3. Plugin API v0 可贡献：
   - 本地工具
   - `ToolSource`
   - Hooks
   - Provider 预设
   - 模型能力数据
4. 提供生命周期：加载、诊断、启用、禁用和 shutdown。
5. 提供 `mao plugin list` / `mao plugin doctor`；具体命令名实现前再次审查 CLI 一致性。
6. 插件加载失败必须隔离并报告，不得阻止 MAO 无插件启动。
7. 第三方 Python 插件按可信本机代码处理；外部工具优先推荐 MCP 进程边界。

### 明确不做

- 不做在线插件市场。
- 不自动安装模型推荐的插件。
- 不从未知 URL 下载并执行 Python 代码。
- 不承诺第三方插件沙箱；Python 插件与 MAO 进程权限相同。

### 发布门

- 至少一个独立示例插件在 wheel 安装环境中成功发现、启用、执行和关闭。
- 不兼容 API 版本被明确拒绝。
- 插件异常不会破坏 Session、工具注册表或无插件启动。
- 插件权限在 CLI/Web 中可见，并需要用户显式启用。

## 6. `v0.2.0` 进入条件

完成 `beta.3` 至 `beta.6` 后，不自动发布 `v0.2.0`。还需满足：

- 至少 10 个外部用户完成安装，至少 5 个完成真实项目任务。
- 没有未解决的 P0 安全、数据损坏或越权问题。
- Provider 兼容矩阵和真实基准可以公开复现。
- Plugin API v0 有明确兼容策略。
- README、英文快速开始、升级说明和迁移说明完成。

## 7. 版本执行纪律

- 开始一个版本时只把该版本第一项标记为 `in_progress`。
- 每个任务记录实现文件、测试命令、结果和剩余风险。
- 版本号只在准备候选包时修改，不在规划阶段提前修改。
- 真实付费 Provider 调用必须由所有者提供本地配置并确认费用边界。
- Tag、Release 和仓库设置变更必须单独确认。
