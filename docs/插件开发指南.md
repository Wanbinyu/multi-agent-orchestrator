# MAO 插件开发指南

本指南说明如何为 MAO 编写一个 Plugin API v0 插件。MAO 通过标准 Python entry point 发现插件，插件默认关闭，用户显式启用后才加载。

完整可运行示例见 [`examples/plugins/mao_wordcount_plugin`](../examples/plugins/mao_wordcount_plugin)。兼容规则见 [`Plugin-API兼容策略.md`](Plugin-API兼容策略.md)。

## 1. 最小插件

一个插件是一个实现 `Plugin` 协议的类，加一个 `create_plugin()` 工厂：

```python
# my_plugin/__init__.py
from src.plugins.api import (
    CAP_TOOLS, MAO_PLUGIN_API_VERSION, PERM_READ_FILES,
    Plugin, PluginContext, PluginManifest,
)
from src.tools.tool_result import ToolResult


def hello(text: str = "", base_dir: str = ".") -> ToolResult:
    return ToolResult(success=True, output=f"hello, {text or 'world'}")


class HelloPlugin:
    def __init__(self):
        self.manifest = PluginManifest(
            id="my-hello",                # 唯一 id：小写字母/数字/连字符
            name="Hello",
            version="0.1.0",
            mao_api_version=MAO_PLUGIN_API_VERSION,  # "0.1"
            description="一个最小示例插件",
            capabilities=[CAP_TOOLS],     # 声明能力
            permissions=[PERM_READ_FILES], # 声明权限（仅展示，非沙箱）
            source="my-hello-plugin",
        )

    def load(self, ctx: PluginContext) -> None:
        ctx.register_tool(
            hello, name="hello", description="打招呼",
            params={"text": {"type": "string", "description": "名字"}},
            category="read",
        )

    def shutdown(self) -> None:
        pass


def create_plugin() -> Plugin:
    return HelloPlugin()
```

## 2. 声明 entry point

在你的包 `pyproject.toml` 中声明 `mao.plugins` entry point，指向工厂函数：

```toml
[build-system]
requires = ["setuptools>=61"]
build-backend = "setuptools.build_meta"

[project]
name = "my-hello-plugin"
version = "0.1.0"
requires-python = ">=3.11"
license = "MIT"

[project.entry-points."mao.plugins"]
hello = "my_plugin:create_plugin"

[tool.setuptools]
packages = ["my_plugin"]
```

本插件依赖 MAO 的 `src` 包；请在已安装 MAO 的环境中安装（MAO 尚未发布到 PyPI）。

## 3. 安装、启用、诊断

```bash
pip install ./my_plugin
mao plugin list                 # 应见到 my-hello
mao plugin enable my-hello      # 写 config/plugins.yaml，下次启动加载
mao plugin doctor               # 诊断发现/兼容/加载健康
mao                             # 启动后 hello 工具可用
mao plugin disable my-hello     # 禁用
```

## 4. PluginContext 贡献能力

`load(ctx)` 中可调用（需与 manifest `capabilities` 对应）：

| 方法 | 能力 | 说明 |
|---|---|---|
| `ctx.register_tool(fn, *, name, description, params, category)` | `tools` | 注册本地工具，`category` 为 `read`/`write`/`execute`/`external`/`unsafe` |
| `ctx.add_tool_source(source)` | `tool_source` | 挂载实现 `ToolSource` 协议的外部工具源（如自建 MCP 适配器） |
| `ctx.add_pre_hook(fn)` / `ctx.add_post_hook(fn)` | `hooks` | 工具执行前/后钩子 |
| `ctx.register_provider_preset(key, preset)` | `provider_preset` | 贡献 WebUI Provider 预设 |
| `ctx.register_model_capabilities(alias, data)` | `model_capabilities` | 贡献模型能力数据（v0 仅记录，未合并入目录） |

`ctx` 记录插件的所有贡献；加载失败或禁用时 MAO 调用 `ctx.rollback()` 撤销，无需插件自行注销。

## 5. 生命周期与隔离

- `create_plugin()` 应轻量，只构造对象与 manifest，不做副作用。
- `load(ctx)` 只做注册；重资源（后台线程、网络连接、子进程）在 `shutdown()` 清理。
- `load` 抛错会被 MAO 捕获，`rollback` 撤销已注册部分，记为诊断；**不阻塞其他插件或无插件启动**。
- `shutdown` 由 MAO 在进程退出或禁用时调用；best-effort，不应抛错。

## 6. 权限与安全模型

- manifest `permissions` 取值：`read_files`、`write_files`、`execute`、`network`。
- **Python 插件是可信本机代码，与 MAO 进程拥有相同权限**。权限列表是给用户看的同意面，**不构成沙箱**；MAO 不会在技术上限制插件能力。
- 用户通过 `mao plugin enable` 显式同意加载；权限在 `mao plugin list` 与 Web「插件」标签可见。
- 需要进程边界隔离的外部工具，优先实现为 MCP 工具源（`tool_source` 能力），而非直接在插件进程内执行不可信代码。

## 7. manifest 校验

`PluginManifest` 在构造时校验：

- `id` 匹配 `^[a-z][a-z0-9-]*$`。
- `name`、`version`、`mao_api_version` 非空。
- `capabilities`、`permissions` 必须是已知常量子集。
- `mao_api_version` 不在 MAO `SUPPORTED_API_VERSIONS` 时，插件被拒绝加载（记 `plugin_api_incompatible`）。

## 8. 调试

- `mao plugin doctor` 用临时注册表做 dry-run，不影响运行中的工具注册表；它会报告发现错误、API 不兼容与加载失败。
- 插件 `load` 中的异常文本不进入公开诊断（脱敏），但 `doctor` 会显示错误码与操作建议；本地开发时可临时在 `load` 内 `print` 或写日志。
- 同 id 插件去重：同 id 多个发行包只保留第一个发现的。

## 9. 不要依赖的内部实现

插件应只通过 `PluginContext` 公共方法贡献。不要直接操作 `tool_registry._tools`、`HookRegistry._pre`、`UI_PROVIDER_PRESETS` 等内部结构--这些不在兼容承诺内，随时可能变。详见 [`Plugin-API兼容策略.md`](Plugin-API兼容策略.md) §1。
