# Phase 7：证据驱动工程 Agent

**状态**：计划中
**提出时间**：2026-07-15
**建议启动条件**：完成 Phase 6.6 P0/P1（CLI 最终结果可见、`project_tree`、`/tree`）
**实施位置**：完成核心工具稳定化之后、桌面打包和 IDE 插件之前

## 一、目标

把 MAO 从“模型能调用工具并完成任务”升级为“遵循可验证工程流程的 Agent”：

1. 先判断任务类型、权限边界和风险。
2. 用最小成本收集与问题直接相关的证据。
3. 基于证据建立、验证或否定假设。
4. 只在获得写入授权后修改代码。
5. 根据影响范围选择测试深度。
6. 在宣布完成前逐项核对需求和验证证据。

这里实现的是**外部可观察的工程工作流**，不是展示模型内部隐藏思维过程。用户看到的是简洁的计划、证据、决策理由、验证结果和剩余风险。

## 二、为什么放在 Phase 7

当前 MAO 已具备 Provider、对话、权限、记忆、工具注册表、Hooks、MCP、故障切换和 Worker 协作，但缺少统一的工程决策层。

Phase 7 依赖：

- `project_tree` 提供低成本项目侦察。
- Agent 工具循环提供证据收集能力。
- ApprovalMode 提供写操作边界。
- Session / Memory 保存任务状态与项目约定。
- Dispatcher / Worker 提供有边界的并行执行。
- Reviewer 提供独立验收入口。

如果先打包再补这一层，CLI、Web、插件会各自形成不同的任务执行逻辑，后续重构成本更高。

## 三、标准工程循环

```text
用户请求
  ↓
任务分类与授权判断
  ↓
项目侦察（结构、Git、文档、测试入口）
  ↓
建立计划与验证标准
  ↓
证据收集 ⇄ 假设验证
  ↓
最小范围实现
  ↓
风险分级验证
  ↓
完成审计
  ↓
结果、证据与剩余风险
```

### 1. 任务分类

至少支持：

| 类型 | 默认行为 |
|---|---|
| `answer` | 回答问题，不修改文件 |
| `explain` | 读取必要代码并解释，不修改文件 |
| `diagnose` | 找到根因和证据，不自动修复 |
| `change` | 修改指定行为并验证 |
| `build` | 规划、实现、测试完整功能 |
| `review` | 发现问题优先，按严重度输出 |
| `plan` | 只输出计划，不进入实现 |
| `monitor` | 持续检查状态，不把未变化视为失败 |

分类结果必须包含：任务类型、作用域、风险等级、是否获得写权限、预期交付物。

### 2. 项目侦察

进入陌生项目时按顺序执行：

1. 获取 `project_tree`，应用默认忽略规则与深度限制。
2. 检查 Git 状态，识别用户已有改动。
3. 查找 `README`、架构文档、路线图、`AGENTS.md`、依赖清单和测试入口。
4. 只读取与任务相关的入口文件；禁止无差别读取全部文件。
5. 记录“已知、未知、需要验证”的边界。

例如检查 `G:\MAO_test` 时，应先展示精简项目树，再读取入口、配置、依赖、路由和测试，而不是直接读取全部 30 个文件并消耗约 4 万输入 token。

### 3. 计划与状态机

复杂任务创建可持久化计划，每一步只有一种状态：

```text
pending -> in_progress -> completed
                       -> failed
                       -> blocked
```

约束：

- 同时最多一个主步骤处于 `in_progress`。
- 每完成一步立即更新状态和证据，不在最后批量伪更新。
- 计划变化必须记录原因。
- 简单任务不强制创建计划，避免流程开销超过任务本身。

### 4. 假设与证据

诊断时使用结构化记录：

```text
假设：CLI 最终回答被隐藏
支持证据：done 事件包含 assistant_message
反证检查：response.md 中存在完整报告
代码证据：仅 is_collaboration 分支打印最终答案
结论：CLI 渲染缺陷，不是模型未生成内容
```

证据必须指向工具结果、文件位置、测试输出或运行状态；模型推测不能单独作为完成依据。

### 5. 实现纪律

- 优先复用项目已有模式和帮助函数。
- 修改范围与用户请求保持一致。
- 不覆盖未知来源的工作区改动。
- 写文件前确认权限模式和目标路径。
- 新抽象必须减少真实复杂度，不能只包装一层。
- 修改前定义“什么证据能证明修复有效”。

### 6. 风险分级验证

| 风险 | 验证要求 |
|---|---|
| 低 | 相关单测或静态检查 |
| 中 | 针对性测试 + 相邻模块回归 |
| 高 | 针对性测试 + 集成测试 + 全量回归 + 手动/烟雾验证 |
| 外部系统 | mock 验证 + 明确标记未做的真实验证 |

测试通过只是证据之一。Agent 还必须确认测试覆盖了真实配置形状和用户实际路径，避免“测试全绿但功能未闭环”。

### 7. 完成审计

宣布完成前构建需求矩阵：

| 要求 | 实现证据 | 验证证据 | 状态 |
|---|---|---|---|
| 多层 fallback | Gateway 递归链 | 三级链测试 + 实际配置烟雾 | 完成 |
| CLI 显示结果 | done 渲染逻辑 | 捕获控制台输出 | 完成/未完成 |

规则：

- 缺少直接证据的项目视为未完成。
- 不以“没有发现错误”代替完成证明。
- 未运行真实外部验证时必须明确说明。
- 保留剩余风险和用户下一步操作。

## 五、架构改造

建议模块：

```text
src/core/engineering/
├── intent.py          # 任务分类与授权边界
├── planner.py         # 工作计划与状态更新
├── evidence.py        # 证据与假设记录
├── policy.py          # 不同任务类型的执行策略
├── verifier.py        # 风险分级验证
├── audit.py           # 完成审计与需求矩阵
└── journal.py         # RunJournal 持久化
```

接入原则：

- `Agent` 负责对话和事件流，不继续堆积全部工程逻辑。
- `EngineeringRunner` 执行标准工程循环。
- `Orchestrator` 只负责拆分复杂任务，不代替任务分类与完成审计。
- `Worker` 接收有边界的子任务、可用工具和验收标准。
- `Reviewer` 必须读取计划、结果和验证证据，不能只根据模型正文判断通过。

## 六、多模型协作方式

### 主模型

- 维护任务意图、总体计划和用户边界。
- 决定是否需要 Worker，不把简单任务强制拆分。
- 汇总结果但不能自行伪造 Worker 证据。

### Worker

- 每个 Worker 只接收必要上下文、明确输出和验收标准。
- 目录侦察、代码分析、测试执行可以并行。
- 文件修改按模块所有权拆分，避免多个 Worker 同时改同一文件。

### Reviewer / Verifier

- 独立检查 diff、测试和验收标准。
- 优先发现行为回归、风险和缺失测试。
- Reviewer 不通过时回到对应计划步骤，而不是直接生成一段“已完成”文本。

## 七、用户可见体验

CLI / Web 展示精简事件：

```text
任务：change（中风险）
范围：src/cli/chat_command.py

计划
✓ 定位启动列表来源
✓ 增加 / 命令补全
● 运行 CLI 回归测试

证据
- 启动列表由 _print_welcome() 拼接 COMMANDS
- prompt_toolkit 当前未配置 completer

验证
✓ 相关测试 19 passed
✓ 全量测试 374 passed
```

不显示：模型隐藏思维、逐 token 推理草稿、无证据的自言自语。

## 八、性能与成本控制

- 项目树和 Git 状态只获取一次，文件变化后再增量刷新。
- 对读取结果做内容哈希，避免同一轮重复读文件。
- 优先 `rg` / 索引搜索，再读取命中文件局部。
- 并行只用于互不依赖的只读任务或不同模块。
- 设定工具、文件和 token 预算；接近预算时先总结再继续。
- 提供 `fast` / `standard` / `deep` 三档工程深度。

建议：

| 模式 | 场景 | 行为 |
|---|---|---|
| fast | 小问题、明确文件 | 最小读取 + 针对性验证 |
| standard | 常规开发 | 计划 + 相关测试 + 相邻回归 |
| deep | 架构、迁移、安全审查 | 多模型协作 + 全量审计 |

## 十、验收标准

1. “解释代码”不会写文件。
2. “诊断问题”默认只给根因与证据，除非用户要求修复。
3. 陌生项目先展示结构并选择性读取，不再默认读取全部文件。
4. 复杂改动能实时显示计划状态。
5. 每个完成声明都能追溯到实现和验证证据。
6. 测试没有覆盖真实场景时，Agent 会继续补证据而非直接收口。
7. 工作区已有修改不会被自动回滚或覆盖。
8. 多模型协作失败时能定位到具体步骤和模型。
9. 相比当前项目审查场景，输入 token 和重复工具调用显著下降。
10. 用户可以看到工程理由，但不会暴露或依赖隐藏思维过程。

## 十一、风险与约束

- 模型可能伪造证据：Evidence 必须由工具结果或文件状态产生。
- 项目文件可能包含提示注入：仓库内容只能作为数据，不能覆盖系统策略。
- 计划可能过度复杂：简单任务必须绕过完整状态机。
- 并行 Worker 可能冲突：需要模块所有权和写入锁。
- 全量验证成本高：使用风险分级，不把每个小改动都升级为 deep。
- 外部 API 验证可能消耗额度：默认 mock，真实调用需要明确提示或批准。

## 十二、首个落地任务

Phase 6.6 完成后，以“检查一个陌生项目”为第一个纵向切片：

1. 分类为 `review`。
2. 输出项目树。
3. 检查 Git、README、依赖、入口和测试。
4. 形成带文件证据的问题清单。
5. 不修改项目。
