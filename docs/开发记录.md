## 2026-07-09

### 问题 1：Windows 下 `Alt+Enter` 触发全屏，无法结束多行输入

**状态**：✅ 已修复

**现象**

运行 `python run.py setup` 进入配置向导，在"系统提示词"步骤出现提示：

```text
(Finish with 'Alt+Enter' or 'Esc then Enter')
```

用户按 `Alt+Enter` 后，Windows 终端直接切换为全屏模式，并未提交输入。

**原因**

`questionary` 的多行输入控件默认使用 `Alt+Enter` 作为提交快捷键。但 Windows 终端（以及多数 Windows 窗口）把 `Alt+Enter` 占用为"全屏切换"快捷键，导致快捷键冲突。

**解决方案**

1. **临时绕过**：按 `Esc` 后再按 `Enter` 提交。
2. **代码修复**：将 `setup_wizard.py` 中的 `multiline=True` 改为 `multiline=False`，即单行输入模式。系统提示词如果需要换行，用户可手动输入 `\n` 或在配置文件里编辑。

**相关文件**

- `src/cli/setup_wizard.py`

### 问题 2：添加 `setup` 子命令后，`run` 不再默认执行

**状态**：✅ 已修复

**现象**

用户输入：

```bash
python run.py 帮我写一个虐恋的小说，在放在E:\小说中，先生成大纲
```

报错：

```text
Usage: run.py [OPTIONS] COMMAND [ARGS]...
No such command '帮我写一个虐恋的小说...'.
```

**原因**

`run.py` 原本只有一个 `run` 命令，用户可以直接把需求当参数。后来新增了 `setup` 子命令，Typer 变成多命令模式，`run` 也需要显式写出：

```bash
python run.py run "需求"
```

这对用户体验不友好。

**解决方案**

在 `if __name__ == "__main__"` 处检测命令行参数：

- 如果第一个参数不是已知的子命令（如 `setup`、`--help`、`-h`），则自动在参数列表前插入 `run`。

修复后用户可直接输入：

```bash
python run.py "帮我写一个虐恋的小说" --output "E:\小说"
```

同时仍支持显式调用：

```bash
python run.py run "需求"
python run.py setup
```

**相关文件**

- `run.py`

### 问题 4：火山方舟 Coding Plan 返回 404 `InvalidEndpointOrModel.NotFound`

**状态**：✅ 已修复

**现象**

运行：

```bash
python run.py "帮我写一个虐恋的小说，放在E:\小说中，先生成大纲" --output "E:\小说"
```

Orchestrator 调用 `glm-ark` 时报错：

```text
Error code: 404 - {'error': {'code': 'InvalidEndpointOrModel.NotFound',
'message': 'The model or endpoint ark-code-latest does not exist or you do not have access to it.'}}
```

用户通过 CCswitch 使用同一模型完全正常，CCswitch 配置如下：

```json
{
  "ANTHROPIC_BASE_URL": "https://ark.cn-beijing.volces.com/api/coding",
  "ANTHROPIC_AUTH_TOKEN": "<YOUR_ARK_API_KEY>",
  "ANTHROPIC_MODEL": "ark-code-latest"
}
```

**原因**

火山方舟 Coding Plan 提供的是 **Anthropic 兼容协议** 接口，而不是 OpenAI 兼容协议接口。必须同时满足两点：

1. **base_url 必须是 `https://ark.cn-beijing.volces.com/api/coding`**，而不是 `/api/v3`。
2. **provider 类型必须是 `anthropic`**，而不是 `openai`。

原来的配置把 Ark 当成了 OpenAI 兼容 provider，用 `/api/v3` 去请求 `ark-code-latest`，所以 Ark 找不到该模型/接入点。

**解决方案**

修改 `config/providers.yaml`：

```yaml
ark:
  name: 火山方舟 Coding Plan
  type: anthropic                              # ← 改为 anthropic
  base_url: https://ark.cn-beijing.volces.com/api/coding  # ← 改为 /api/coding
  api_keys:
    - ${ARK_API_KEY}

models:
  glm-ark:
    provider: ark
    model_id: ark-code-latest

  kimi-for-coding:
    provider: ark
    model_id: kimi-for-coding
```

其中：

- `ARK_API_KEY` 的值就是 CCswitch 里的 `ANTHROPIC_AUTH_TOKEN`。
- `ark-code-latest` 是 Coding Plan 的共享路由模型，实际后端为 GLM 5.2。
- `kimi-for-coding` 是 Coding Plan 中对应 Fable 角色的路由模型。

**相关文件**

- `config/providers.yaml`
- `.env`

### 问题 5：子工程师使用 `kimi-for-coding` 返回 `UnsupportedModel`

**状态**：✅ 已修复

**现象**

生成过程中，子工程师 `story_speak` 调用 `kimi-for-coding` 时报错：

```text
Error code: 404 - {'error': {'code': 'UnsupportedModel',
'message': 'The requested model does not support the coding plan feature. Please refer to the documentation at
https://www.volcengine.com/docs/82379/1925114 to select a compatible model.'}}
```

**原因**

火山方舟 Coding Plan Pro 套餐**不支持 `kimi-for-coding` 模型**。该模型可能需要更高级别的套餐或单独开通。用户的套餐只支持 `ark-code-latest` 等 Coding Plan 内模型。

**解决方案**

将所有使用 `kimi-for-coding` 的地方改为 `glm-ark`（即 `ark-code-latest`），这是用户套餐支持的模型。

修改 `config/workers.yaml`：

```yaml
orchestrator:
  model: glm-ark
  system_prompt: |
    可用 Worker 列表：
    - story_speak: 故事大师，默认模型 glm-ark

available_workers:
  story_speak:
    name: 故事大师
    default_model: glm-ark          # ← 从 kimi-for-coding 改为 glm-ark
```

同时 `reviewer.model` 也从 `claude-sonnet-5` 改为 `glm-ark`，避免同样问题。

另外，在 `config/providers.yaml` 的 `kimi-for-coding` 模型下增加注释，提示部分套餐不支持：

```yaml
kimi-for-coding:
  provider: ark
  # 注意：部分 Coding Plan 套餐（如 Coding Plan Pro）可能不支持该模型，如遇 404 请改用 glm-ark
  model_id: kimi-for-coding
```

**经验总结**

通过 Ark Coding Plan 接入时， safest 的做法是**所有 Worker 和 Orchestrator 都统一使用 `glm-ark`（ark-code-latest）**，除非确认当前套餐支持其他模型。

**相关文件**

- `config/workers.yaml`
- `config/providers.yaml`

### 问题 6：Kimi 官方 API 配置错误导致 404

**状态**：✅ 已修复

**现象**

用户想接入 Kimi 官方 API，最初提供的信息：

- 请求链接：`https://api.kimi.com/coding/`
- Key：`sk-kimi-...`

配置后运行报错：

```text
Error code: 404 - {'error': {'message': 'The requested resource was not found',
'type': 'resource_not_found_error'}}
```

**原因**

用户实际在 CCswitch 中使用的并不是 `api.kimi.com`，而是另一个 Anthropic 兼容转发服务：

```json
{
  "ANTHROPIC_BASE_URL": "https://api.va11.icu/",
  "ANTHROPIC_AUTH_TOKEN": "<YOUR_KIMI_API_KEY>",
  "ANTHROPIC_MODEL": "kimi-for-coding",
  "ANTHROPIC_DEFAULT_SONNET_MODEL": "kimi-for-coding",
  "ANTHROPIC_DEFAULT_OPUS_MODEL": "kimi-for-coding",
  "ANTHROPIC_DEFAULT_HAIKU_MODEL": "kimi-for-coding",
  "ANTHROPIC_DEFAULT_FABLE_MODEL": "kimi-for-coding"
}
```

所以原来的 `base_url` 和 `model_id` 都配置错了。`api.kimi.com/coding/` 无法识别 `kimi-for-coding` 这个 model id。

**解决方案**

修改 `config/providers.yaml`：

```yaml
kimi:
  name: Kimi 转发服务（via CCswitch）
  type: anthropic              # ← 不是 openai
  base_url: https://api.va11.icu/
  api_keys:
    - ${KIMI_API_KEY}

models:
  kimi-for-coding:
    provider: kimi
    model_id: kimi-for-coding   # ← CCswitch 中所有 Anthropic 模型名都映射到这个模型
```

修改 `.env`：

```env
KIMI_API_KEY=<YOUR_KIMI_API_KEY>
```

同时 `config/workers.yaml` 中所有默认模型改为 `kimi-for-coding`。

**经验总结**

1. 很多"Kimi 接入"其实是通过 Anthropic 兼容转发服务实现的，provider 类型要选 `anthropic`。
2. 一定要以**实际能跑通的 CCswitch 配置**为准，而不是用户口头描述的地址。
3. 转发服务通常会把所有 Anthropic 模型名（`claude-sonnet`、`claude-opus` 等）映射到同一个底层模型。
4. API Key 只应保存在 `.env` 中，文档和聊天记录里都应使用占位符。

**相关文件**

- `config/providers.yaml`
- `config/workers.yaml`
- `.env`


```bash
cd E:\multi-agent-orchestrator

python run.py "你的需求" --output "E:\小说"
```
